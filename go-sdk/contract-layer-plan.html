go-stellar-sdk · contract-layer plan

A high-level contract layer for the Go SDK.

The JS SDK already turns a Soroban contract call into a handful of lines. Go does not — moving value still means 30–50 lines of hand-rolled host-function plumbing. The plan is to close that gap by porting the JS SDK's high-level abstractions into idiomatic Go. This page is the map: where we are headed, and what each PR delivers along the way.

Approach
Port the JS SDK contract abstractions, idiomatically & additively
Goal
A token send in ~5 lines, on parity with JS

01The idea: port the abstractions without reinventing them

Through the Go SDK, a transfer currently takes a few steps: building a raw xdr.HostFunction, simulating, reading back the footprint and resource fee, re-simulating (optional), signing each auth entry, submitting through rpcclient, then polling. The same transfer is about five lines in @stellar/stellar-sdk (the JS SDK), which wraps those steps in a high-level layer Go doesn't offer yet.

The JS SDK achieves it with a small set of high-level abstractions that sit on top of the raw XDR/RPC surface and hide the state machine. They live in its contract module. We are porting those same abstractions to Go — idiomatically (contexts, functional options, errors.Is/As, channels over callbacks), and strictly additively, so nothing existing changes.

The four abstractions we are porting

A contract.Client Generic invoke against any Soroban contract. The kernel everything else delegates to.
B AssembledTransaction The build → simulate → sign → send → poll lifecycle, made an explicit, ctx-driven object instead of a hidden sequence.
C SentTransaction A submitted transaction you can Wait() on or Watch() as a channel of status updates.
D Spec (optional) XDR ⟷ native-Go marshaling, ported from the JS spec.ts allowing callers to pass Go values, not ScVals.
+ on top Token A SEP-41 surface built on contract.Client — the ergonomic token API the goal below uses.

02The goal: a token send in five lines

Here is where we are headed. One surface, the same five lines whether you are sending native XLM, a credit asset, or an SAC — it auto-routes classic vs. Soroban underneath so application code never has to ask "is this a G… or a C…?" 

// The headline. Same five lines for XLM, USDC, or a pure SAC.
tok, _  := asset.New(xdr.MustNewNativeAsset(),
    asset.WithRPC(rpc), asset.WithNetwork(network.TestNetworkPassphrase),
    asset.WithSource(kp.Address()), asset.WithSigner(contract.KeypairSigner(kp)))

at, _   := tok.Transfer(ctx, kp.Address(), recipient, big.NewInt(10_000_000))
sent, _ := at.SignAndSend(ctx, contract.KeypairSigner(kp))
_, _     = sent.Wait(ctx)

That is the destination. We don't ship it on day one — we ship the kernel it rests on first, then thicken the lifecycle until this code is real. The next section shows the lifecycle those five lines drive; the section after that breaks the road into PRs.

03The shape of one call

Every contract call walks the same lifecycle. AssembledTransaction makes it an explicit object so each stage is something you can hold, inspect, and hand off. The first PR lights up the left half — build → simulate → read — and stops before signing. Every PR after it thickens the dimmed half until the full path is real.

Reading is the thinnest end-to-end path that still proves the whole pipeline composes: xdr → Client → AssembledTransaction → Simulate → decode. It needs no signer and submits nothing — so it is small, self-evidently correct, and every later PR lands on top of it.

PR 1Read a contract
shipping now · in review
10 files · +2,144 · additive

The foundation everything else lands on. contract.Client and AssembledTransaction exist; Invoke runs build → simulate and hands back the assembled transaction. For a view method, IsReadCall() is true and Result() returns the simulated value — nothing is signed and nothing is submitted. Arguments go in as pre-built []xdr.ScVal (eased by the new xdr.Scv* builders), and the result comes back as a raw xdr.ScVal. No signer, no Spec yet — those are the next two PRs, and they bolt on without moving a wall. 

// Build a client — no network touched yet.
c, _ := contract.New(contractID, rpc, network.TestNetworkPassphrase,
    contract.WithSource(account.Address()))

// Invoke a view method: build the host fn, then simulate it.
at, _ := c.Invoke(ctx, "balance", []xdr.ScVal{holder})

if at.IsReadCall() {    // no address-credentialed auth, no read-write footprint
    res, _ := at.Result()       // raw xdr.ScVal — PR 2 will type-decode this
    scv := res.(xdr.ScVal)
}
PR 2Typed arguments & results
+ Spec, marshaling,
method validation

Layer the Spec abstraction on top. The same Invoke now accepts native Go values as a map[string]any (today it only takes raw ScVals and rejects maps), marshals them against the contract's spec, and decodes the return value to a native Go type. Method names are validated against the spec, with a "did you mean…?" suggester. No raw xdr.ScVal in sight. 

// Same Invoke signature — now hand it native Go values.
at, _ := c.Invoke(ctx, "balance", map[string]any{"id": holder})

bal, _ := at.Result()   // *big.Int — decoded via Spec, not a raw ScVal
PR 3Signing & submission
+ Signer, SignAndSend,
SentTransaction, restore

Thicken the lifecycle past simulate. The Signer interface, SignAndSend, and SentTransaction with Wait() / Watch() complete the write path. The fee ceiling (MaxFee) is enforced before the wire; archived state is auto-restored and re-simulated transparently. This PR also folds in the lifecycle's most error-prone details — the sequence-number handling around simulate and restore — so callers never hit txBadSeq. 

// One call: build → simulate → sign → send → poll.
val, hash, _ := contract.InvokeAndConfirm(ctx, c, "increment",
    map[string]any{}, contract.KeypairSigner(kp))

// …or drive the lifecycle yourself, stage by stage:
at, _   := c.Invoke(ctx, "increment", nil)
sent, _ := at.SignAndSend(ctx, contract.KeypairSigner(kp))
res, _  := sent.Wait(ctx)
PR 4The token surface
the goal · JS parity
+ Token, SEP-41

The marquee. Token is a hand-written, canonical SEP-41 surface built on contract.Client using a bundled SAC spec — no codegen involved. Transfer auto-dispatches: a classic txnbuild.Payment when both parties are accounts and the asset is classic, a Soroban invocation whenever a contract address is involved. This is the PR that closes the JS-parity gap — the five-line send from the goal becomes real, live on testnet. 

// Same five lines for XLM, USDC, or a pure SAC.
tok, _  := asset.New(xdr.MustNewNativeAsset(),
    asset.WithRPC(rpc), asset.WithNetwork(network.TestNetworkPassphrase),
    asset.WithSource(kp.Address()), asset.WithSigner(contract.KeypairSigner(kp)))

at, _   := tok.Transfer(ctx, kp.Address(), recipient, big.NewInt(10_000_000))
sent, _ := at.SignAndSend(ctx, contract.KeypairSigner(kp))
_, _     = sent.Wait(ctx)
PR 5Multi-party signing
+ auth hand-off
ToXDR / FromXDR

Some contracts require signatures from someone other than the transaction's source. This PR adds cross-host coordination: NeedsNonInvokerSigningBy() reports who still owes a signature, and ToXDR / FromXDR carry the in-flight transaction and its simulation between processes so a co-signer can sign just the auth entries and hand it back. 

// Coordinator simulates, sees a co-signer is owed, ships the tx out.
at, _   := c.Invoke(ctx, "transfer", args)
owed    := at.NeedsNonInvokerSigningBy()   // [G...cosigner]
blob, _ := at.ToXDR()                       // hand off to the other host

// Co-signer's host: rebuild, sign only the auth entries, return it.
at2, _ := contract.FromXDR(ctx, blob, rpc, net)
at2.SignAuthEntries(ctx, cosigner)
PR 6Receiving events
+ contract/events
decoders + Stream

The receive side. A contract/events package decodes SEP-41 and SAC events into typed Go values, validates SAC event integrity (to reject impostor contracts), and streams them off RPC. It depends only on the XDR decode primitives, so it can land any time after PR 1 — independent of the send lifecycle. 

// Decode and stream typed SEP-41 / SAC events off RPC.
for ev := range events.Stream(ctx, rpc, contractID) {
    if t, ok := ev.(events.Transfer); ok {
        fmt.Println(t.From, "→", t.To, t.Amount)
    }
}
PR 7Codegen · optional
+ cmd/sorobangen
typed bindings

For arbitrary contracts (not just tokens), a code generator emits a per-contract typed Go client from a WASM or spec — typed methods, UDT structs, typed errors, and an init() that registers the spec and event decoders. Each generated client embeds a *contract.Client and forwards to Invoke. It is on nobody's critical path; token users never touch it. 

# Generate a typed client from a contract's WASM:
stellar contract bindings go --wasm token.wasm --out ./gen/token
import _ "myapp/gen/token"   // init() registers spec + event decoders

client, _ := token.NewClient(contractID, rpc, net, token.WithSource(kp.Address()))
bal, _    := client.Balance(ctx, holder)   // typed method, typed return — no ScVals

Reference implementation of all seven PRs already exists on a fork and runs end-to-end on testnet (native + SAC transfers, multi-party signing, event streaming, codegen), so these milestones are de-risked rather than speculative. Full design rationale lives in the RFC "A high-level Soroban contract layer for the Go SDK"; a detailed walkthrough of PR 1 lives in "Reading a contract, end to end." Each PR is purely additive — zero renames, zero removals in xdr / txnbuild / rpcclient / keypair.
