Benchmarking Stellar RPC v2 — what's the actual problem?

RPC v2 swaps a 7-day SQLite window for a full-history engine. Before it ships, the team must prove it meets its SLAs and keep proving it on every change — and a benchmark number only means something if everything except the code is held fixed.

01The system under test

v1 stores ~7 days of ledgers in SQLite. v2 stores the entire chain (60M+ ledgers, tens of billions of events, ~1.3 TB) in two tiers — and every tier × data type (ledgers / transactions / events) is a surface to benchmark.

RPC v1 — today

SQLite, retention window

One DB transaction per ledger; data older than ~7 days is trimmed. Simple, but can't serve history. The baseline to beat.

02Why it's hard: the measurement equation

Every result is a function of four inputs. To attribute a change to the code, the other three must be bit-identical between runs — and each one is independently expensive to pin.

result = f(code, dataset, machine, starting DB state )
VARY

Code

The PR under test. The only thing allowed to differ between a run and its baseline.

MUST PIN

Dataset

Real pubnet ledgers are whatever the network happened to produce — not controllable, not reshapeable (can't dial up Soroban load or event density).

MUST PIN

Machine

Cross-machine runs (c6id.2xl → 8xl, x86 vs Graviton) showed large swings in throughput & latency. Numbers from random CI runners are noise.

MUST PIN

Starting state

Ingesting into an empty DB performs nothing like ingesting into one holding a week+ of history. Each run needs the same pre-filled DB, unmutated.

The crux: pinning each variable has a real cost — generating deterministic data, reserving fixed hardware, restoring a multi-hundred-GB database per run — and it all has to fit a CI cadence and budget. The roadmap calls this out as risk #2: “something we've never built … the approach is unproven.”

04The constraints that kill naive designs

⏱ CI-speed
Must not run (much) longer than the other integration tests it sits beside.
💰 Budget
No $540/mo FSR snapshots, no io2 IOPS bills. Volume clones ≈ pay only while testing.
🎯 Representative
NVMe would give consistent numbers — but operators run GP3; its burstiness is the workload. Consistency ≠ relevance.
♻️ Fresh state every run
No tombstones, no copy-on-write artifacts, no test ledgers accumulating — each run starts from the identical golden DB.
