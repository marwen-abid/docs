Stellar RPC,
from the inside.

A JSON-RPC server that exposes the Stellar network's current view — submit transactions, query their status, read ledger entries, and simulate Soroban smart-contract calls. It runs a Captive Core subprocess, ingests every closed ledger into a local SQLite store, and answers queries from that store plus live Core lookups.

The one thing to internalize first: this is a Go daemon that links two Rust static libraries over cgo. That hybrid shapes the build, the simulation engine, and the most interesting design in the repo.

01

The 10,000-ft view

Strip away the detail and the daemon does just two jobs, plus a sandbox:

① Ingest & store

A background service pulls each newly-closed ledger from Captive Core and writes it — ledger header, transactions, events — into local SQLite, one DB transaction per ledger. Old data is trimmed to a retention window. This is the daemon's “memory.”

② Serve queries

A JSON-RPC server answers reads (getEvents, getLedgers, getTransaction(s), getFeeStats…) from that SQLite store, plus live state lookups (getLedgerEntries) straight from Core.

③ Simulate (the sandbox)

simulateTransaction runs a Soroban host function in a Rust sandbox over cgo, fetching only the ledger entries it actually touches. It returns the resource footprint, fees, auth, and predicted state changes a client needs to build a real transaction.

Everything else is supporting cast: configuration, a Captive Core subprocess as the ledger source, in-memory windows for fee statistics, request throttling middleware, an optional cloud-datastore backend for serving history beyond local retention, and the Rust FFI libraries that power simulation and XDR→JSON conversion.

The cast of characters

  • Captive Core — a stellar-core subprocess the daemon launches and manages; its source of truth for ledgers and live state. Two HTTP clients talk to it: a normal one (submit, info) and a “fast” one on Core's HTTP query port (ledger-entry reads & simulation).
  • Ingestion service — the loop that turns closed ledgers into rows in SQLite.
  • SQLite store — WAL mode, manual checkpointing, reader/writer split, retention-trimmed.
  • JSON-RPC handlers — one per method, each wrapped in throttling + metrics middleware.
  • Preflight worker pool + Rust libs — the simulation engine, behind a cgo boundary.

04

Daemon lifecycle & wiring

The composition root Go

The cobra CLI plus the Daemon struct that constructs, owns, starts, and gracefully tears down every other subsystem in a fixed order.

This is the place where all the independently-built pieces are instantiated and wired to each other. main.go is a thin CLI shell; the real work is daemon.go's MustNew (build everything, fail-fast via logger.Fatal) and Run (serve until a shutdown signal). The Daemon struct holds a reference to every subsystem and is itself passed — as the narrow interfaces.Daemon abstraction — into the subsystems that need shared resources. Read this first: understanding boot order explains the whole system.

Key files

  • main.goCobra root: run (default), version, gen-config-file.
  • daemon/daemon.goThe Daemon struct, MustNew startup order, all create*/must* helpers, HTTP+admin servers, graceful Close(), storage warm-up.
  • daemon/metrics.goImplements the interfaces.Daemon methods; build-info gauge + the CoreClientWithMetrics submission wrapper.
  • daemon/interfaces/interfaces.goThe decoupling seam: Daemon, CoreClient, FastCoreClient interfaces.
  • util/panicgroup.goServer goroutines that os.Exit(1) on panic after logging a stack.

Key abstractions

Special data structures

Two separate Core HTTP clients
Submit/info go through the normal client; latency-sensitive ledger-entry reads go through a second client on Core's HTTP query port (served from an in-memory snapshot). Surfaced as two interfaces so callers can't confuse them.
Fee-window fill modeled as a DB migration

Where do I look when…
  • …I need to add a subsystem or change start order?
    MustNew (daemon.go:175) + a field on the Daemon struct (daemon.go:52).
  • …I need to change shutdown order?
    Daemon.close (daemon.go:88).
  • …a feature should hit a different Core endpoint?
    createStellarCoreClient / createHighperfStellarCoreClient (daemon.go:304–316).

Gotchas
  • MustNew never returns an error — every failure is logger.Fatal (os.Exit). No partial-startup recovery.
  • Startup order is load-bearing: backfill must finish before live ingestion, and afterwards db.ResetCache() + feewindows.Reset() are called so caches repopulate from the backfilled DB.
  • The Prometheus namespace is still "soroban_rpc" (a rename to stellar_rpc is pending) — relevant when querying metrics.
  • Version/commit/branch/soroban-env strings come from -ldflags at build time; a bare go build reports dev.
05

Configuration

One declaration, three namespaces Go

Every setting is a single Option that simultaneously defines its env var, CLI flag, TOML key, default, parser, and validator.

The whole point is to prevent drift: a setting is described once as a struct literal in options.go, and that one description drives flag registration, env lookup, TOML parse/marshal, and validation. The env var and TOML key auto-derive from the flag name (database-url → DATABASE_URL), so the three can never silently disagree. It also backs the gen-config-file command, which emits a fully-commented default TOML.

Key files

  • config/options.goThe authoritative list — cfg.options() returns every Option literal (~60 settings). This is where you add/change a setting.
  • config/option.goThe Option struct + per-option set/marshal/validate helpers.
  • config/main.goThe Config struct (one field per setting) + SetValues load orchestration + Validate.
  • config/parse.go · flags.go · toml.goType-specific parsers · pflag binding · TOML read/write.
  • config/version.goBuild-info vars overwritten via -ldflags.

Key abstractions

NameWhat it is
Option structName + EnvVar + TomlKey + Usage + DefaultValue + ConfigKey (a typed pointer into the Config field it writes) + optional CustomSetValue/Validate/MarshalTOML hooks.
Config.SetValues funcEstablishes precedence: defaults → TOML file → env → flags (env+flags are applied twice so they override file values).
CustomSetValue hooksEscape hatch for non-trivial settings (log-level, --network shortcut, datastore config).
ConfigKey as a typed pointer into Config
Both the pflag type and the parser are chosen by type-switching on that pointer — a setting's Go type alone determines its parsing/flag behavior. Guard-rail tests enforce every field has exactly one Option pointing at it.
Layered re-application precedence
Env+flags are applied once (to discover the config-file path), the file is loaded, then env+flags are applied again so explicit env/CLI always win over the file.
Where do I look when…
  • …I need to add a config setting?
    Add a field to Config (main.go), then append an Option literal in cfg.options() (options.go). Guard-rail tests fail until it's wired correctly.
  • …I need cross-field validation or a custom type?
    Set the Validate / CustomSetValue hooks (model on log-format, network); add a parser branch in parse.go if needed.
  • …the version string is wrong?
    version.go vars + the GOLDFLAGS/RS_ENV_VERSION block in the Makefile.
Gotchas
  • Precedence is defaults < TOML < env < flags — and env+flags are applied a second time after the file; easy to miss.
  • Some “defaults” are set inside a Validate hook because they depend on other parsed values (e.g. stellar-core-url derives from the captive-core port).
  • gen-config-file calls SetValues but intentionally not Validate, so it can emit a partial template.
  • soroban-env-prev/curr versions come from the Makefile parsing cargo metadata with jq, not from Go.
06

JSON-RPC transport & handler registration

The front door Go

Takes an HTTP POST, runs it through CORS + a 512 KB cap + per-method limiters + metrics, bridges it to a creachadair/jrpc2 dispatcher, and routes the method name to its handler.

NewJSONRPCHandler in internal/jsonrpc.go is the single assembly point. It declares every supported method, builds each handler with its dependencies, wraps each in a middleware chain, then mounts them on a jhttp.Bridge that speaks HTTP on one side and jrpc2 on the other. Cross-cutting concerns (back-pressure, timeouts, metrics, CORS, size cap, logging) live here so the individual method handlers stay focused on business logic.

The 12 registered methods

getHealth · getEvents · getNetwork · getVersionInfo · getLatestLedger · getLedgers · getLedgerEntries · getTransaction · getTransactions · sendTransaction · simulateTransaction · getFeeStats

Key abstractions

NameWhat it is
NewJSONRPCHandler funcBuilds a slice of per-method config (handler, queue limit, duration limit), wraps each into a dispatch handler.Map, and feeds it to the bridge behind global limiters + cap + CORS.
methods.NewHandler funcThe generic pattern every method uses: turns a typed func(ctx, Req)(Resp,err) into a jrpc2 handler and calls AllowArray(false) to force named params.
jhttp.BridgeThe jrpc2 ⇄ HTTP adapter — parses the envelope, dispatches by method name, serializes the reply.
decorateHandlers funcWraps every handler with the Prometheus timing decorator (request_duration_seconds, labeled endpoint + status).
Name-based dispatch map (handler.Map)
JSON-RPC routes by the request's method string, not by URL path — every request hits "/". A map[string]jrpc2.Handler keyed by the method-name constant is the natural structure jrpc2 consumes directly.
AllowArray(false) named-params only
Forcing object params (not positional arrays) means request structs can gain new optional fields (like xdrFormat) without breaking existing clients. A deliberate backward-compatibility choice baked into the generic handler.
bufferedResponseWriter
The HTTP duration limiter must be able to abandon a slow request and return 504 instead of a half-written body, so it buffers the downstream handler's output and only flushes if it finished in time.
Request flow. POST → CORS → MaxBytesHandler (512 KB) → global HTTP duration + backlog limiters → jhttp.Bridge (looks up method) → timing decorator → per-method duration limiter → backlog limiter → the business handler → handler reads DB/Core and emits base64 XDR or (if xdrFormat:"json") JSON via xdr2json → decorator records duration+status → bridge serializes reply.
Where do I look when…
  • …I need to add a new RPC method?
    Append an entry to the handlers slice in NewJSONRPCHandler (jsonrpc.go); add methods/<name>.go returning NewHandler(fn); add the method-name constant + request/response types in the go-stellar-sdk protocols/rpc package; add config limits.
  • …I need to change concurrency / timeout limits?
    The cfg.RequestBacklog*QueueLimit / cfg.Max*ExecutionDuration values in the handlers slice; enforcement in network/.
  • …I need to change CORS / size cap / mount point?
    Bottom of NewJSONRPCHandler; the "/" mount in createHTTPHandler (daemon.go).
Gotchas
  • There are two limiter layers: per-method (inside the bridge) and global (wrapping the whole bridge).
  • An empty xdrFormat is valid and means base64. Only "json" and "base64" are accepted.
  • getTransactions is the exception that uses handler.New directly, so it does not disable array args.
  • A malformed handler signature panics at startup (in handler.Check), not at request time.
07

Read / query methods

Where the query logic lives Go

The read-side handlers that answer getEvents, getLedgers, getTransaction(s), getLedgerEntries, and getFeeStats — each translating cursor-based pagination and XDR-vs-JSON output.

Each methods/get_*.go is a thin handler: validate the request, compute a query range from a pagination cursor, call a db reader (or the fast Core client), and re-encode the resulting XDR into base64 or JSON. The heavy lifting (SQL, XDR decode) lives in internal/db. A recurring theme: responses always echo the store's current ledger window so clients can reason about retention and freshness.

The methods at a glance

MethodReads fromCursor model
getEventsSQLite events table4-part Cursor {ledger, tx, op, event} — also the row primary key
getTransactionsSQLite (replays ledger meta)single int64 TOID string
getLedgersSQLite, then datastore fallbackplain decimal ledger sequence
getTransactionSQLite (hash join)—
getLedgerEntrieslive Core (fast client), not SQLite—
getFeeStatsin-memory fee windows—

Special data structures

Event cursor stored AS the SQLite primary key
The id is a zero-padded <TOID>-<eventIndex> string, so lexicographic comparison equals chronological order. Range queries become WHERE id ≥ start AND id < end ORDER BY id and retention trims become DELETE WHERE id < cutoff — no numeric composite-key gymnastics.
Four discrete topic columns (topic1…topic4)
Soroban events have up to 4 topics. Instead of a join table, each is its own indexed column of marshaled ScVal bytes, so topic filters compile straight to topicN = ? equality predicates, assembled as OR-of-AND clauses.
getLedgers local-then-datastore range split
fetchLedgers splits the requested range at the local first-ledger boundary into all-local / all-remote / split, so a single call can transparently span SQLite + cloud object store.
getEvents flow. validate → derive a start Cursor (from startLedger or the pagination cursor, with Event++) and an exclusive endLedger (start + a 10,000-ledger scan cap, clamped to retention) → assemble contract/type/topic filters → DB scan calls a ScanFunction per row until limit → format EventInfo (XDR or JSON) → next cursor.
Where do I look when…
  • …I need to change pagination / next-cursor logic?
    get_events.go (start/end derivation + cursor emission); get_transactions.go initializePagination; cursor encoding in go-stellar-sdk protocols/rpc/cursor.go.
  • …I need to add/modify an event filter (contract, type, topic)?
    combineContractIDs/EventTypes/Topics in get_events.go + the SQL assembly in db/event.go GetEvents.
  • …I need to change where ledger entries are read from?
    ledgerentries/main.go (coreLedgerEntryGetter); the fast client interface in daemon/interfaces.
  • …I need to tweak XDR↔JSON formatting?
    methods/json.go + each handler's switch format block.
Gotchas
  • getLedgerEntries does not read SQLite — it queries live state from the fast Captive-Core client at the latest ledger. TTL keys are explicitly rejected. NotFound keys are silently dropped, so the response can be shorter than the request.
  • When paginating, the start position is the item after the cursor (Event++ / TransactionOrder++). Forgetting this re-returns the cursor item.
  • getEvents caps each scan at 10,000 ledgers regardless of limit; if fewer than limit events are found, the returned cursor is a window marker (MaxCursor at endLedger-1), not a real event id.
  • getFeeStats reads precomputed percentiles from memory — no SQLite scan per request.
08

Simulate & submit

The only write/execute endpoints Go ⇄ Rust

simulateTransaction preflights a Soroban op against current ledger state via the Rust worker pool; sendTransaction forwards a signed tx to Captive Core and maps its status back.

simulateTransaction takes an unsigned/partial envelope, gathers the ledger context the simulation needs (latest seq, bucket-list size, protocol version), dispatches the single operation to the Rust preflight engine through a bounded worker pool, then assembles the output — resource footprint, minimum fee, return value, required auth, diagnostic events, predicted state changes, and an optional restore preamble — into a response. sendTransaction is much thinner: decode, hash, forward the raw base64 to Core, translate Core's submission status.

Key abstractions

NameWhat it is
PreflightGetter interfaceSingle-method seam (GetPreflight) the handler depends on; satisfied by the worker pool. Keeps the handler testable and decoupled from cgo.
preflight.Preflight structThe decoded simulation output: error, events, transaction data (footprint+fees), min fee, return value, auth, CPU/memory, pre-restore data, and ledger-entry diffs.
formatResponse funcThe single place every simulation field is serialized into the wire response (XDR vs JSON).
validateAuthMode funcInfers the auth mode when omitted: invoke-with-auth → enforce, without → record.
The restore preamble (PreRestore* fields)
State-archival: an entry the tx touches may be archived. Simulation detects this and returns a separate SorobanTransactionData describing a RestoreFootprint operation the client must submit first. The dual-data shape exists because restore is a distinct prerequisite transaction.
LedgerEntryDiff → StateChanges via Before/After presence
Each diff carries optional Before and After entry bytes; the presence pattern (before-only=Deleted, after-only=Created, both=Updated) compactly encodes the change type without an extra enum crossing the FFI boundary.
Bounded job-queue worker pool, fail-fast
Preflight runs untrusted Soroban code on a dedicated 100 MiB-stack thread and is CPU/memory heavy. A fixed-size channel + fixed worker count caps concurrency; a full queue returns ErrPreflightQueueFull immediately rather than queueing unboundedly.
Where do I look when…
  • …I need to change which ops/validation simulate accepts?
    NewSimulateTransactionHandler in simulate_transaction.go.
  • …I need to add/modify a simulate response field?
    formatResponse / getSimulationResults / getRestorePreamble in simulate_transaction.go; the Rust side mirrors it (see §14).
  • …I need to change submission status mapping?
    The status switch in send_transaction.go (ERROR/PENDING/DUPLICATE/TRY_AGAIN_LATER).
  • …I need to tune simulation concurrency?
    createPreflightWorkerPool (daemon.go) + config PreflightWorkerCount / PreflightWorkerQueueSize.
Gotchas
  • simulateTransaction almost never returns a JSON-RPC error — nearly every failure becomes a normal 200 response with the Error string populated. sendTransaction is the opposite (typed jrpc2 errors).
  • Exactly one operation and zero memo are hard requirements for simulation.
  • For ExtendFootprintTtl/RestoreFootprint the caller must supply SorobanTransactionData (footprint can't be inferred); InvokeHostFunction lets simulation compute it.
  • Resource/fee bumping is not done here — the min fee + footprint are computed in Rust and merely relayed; clients apply them before signing.
09

SQLite store

Store the blob, index the rest Go

A WAL-mode SQLite store with a Reader/Writer split that persists raw ledger XDR blobs plus thin index tables, trimmed to a retention window on every commit.

The core storage philosophy: store the raw LedgerCloseMeta (LCM) XDR blob once, index everything else. Rather than denormalizing transactions and events into fully-populated tables, the daemon keeps only lightweight index tables (hash → ledger+position for transactions; id/topics/contract → blob for events) and re-derives full objects on the fly from the stored LCM blob at read time. This keeps the database small enough to fit a sliding retention window.

Key abstractions

NameWhat it is
ReadWriter / WriteTxNewTx() opens one SQLite transaction and yields a WriteTx exposing LedgerWriter/TransactionWriter/EventWriter — all share the same tx so one ledger's rows commit atomically together.
LedgerReader / TransactionReader / EventReaderThe read side. GetTransaction joins the hash table to the LCM and seeks into the blob; GetEvents scans a cursor range.
dbCache structRWMutex-guarded in-memory cache of latest ledger seq + close-time, so getLatestLedger avoids a MAX(seq) query on the hot path.
ParseTransaction funcThe on-the-fly hydration — turns an LCM + tx into a full Transaction with all XDR fields and event lists. “Store the blob, derive the rest,” realized.
Migration frameworkMultiMigration/guardedMigration — data migrations (ledger-range replays, guarded by the metadata table) distinct from SQL schema migrations.

Special data structures

Raw LCM blob as the single source of truth
A ledger's full content is XDR and large; it's stored once as a BLOB keyed by sequence. Transactions/events tables hold only pointers, and full objects are re-derived from the blob at read time. Minimizes storage; costs re-parsing on reads.
Transaction table keyed by 32-byte hash + application_order
getTransaction maps an opaque hash to a position, then joins to the LCM and Seek(application_order-1) instead of decoding the whole ledger. Fee-bumps get TWO rows (outer + inner hash) so either resolves to the same tx.
TOID sentinel positions for non-operation events
Pre-tx, per-op, post-tx and post-all-txs event stages share one cursor scheme by using reserved sentinel slots (before-all = tx 0; post-tx = max op mask; after-all = max tx mask), preserving both chronological sort order and trimmability.
Batched multi-row INSERTs
SQLite caps bind variables at ~32,766. Transactions (3 vars/row) batch at 3,000 rows; events (10 vars/row) at 1,000 — so a large ledger never exceeds the limit.
Write (per ledger). NewTx → InsertLedger (the blob) + InsertTransactions (hash-index rows, 2 for fee-bumps) + InsertEvents (TOID cursor ids + topics) → Commit: trim everything older than the retention window → COMMIT + atomically bump dbCache → PRAGMA wal_checkpoint(TRUNCATE).
Where do I look when…
  • …I need to add a table/column/index?
    Add a numbered file in db/sqlmigrations/ — auto-embedded via //go:embed and applied at startup.
  • …I need to change retention/trimming?
    trimLedgers/trimTransactions/trimEvents, all called from writeTx.Commit.
  • …I need a one-time backfill of an index table?
    Register a migration in BuildMigrations (migration.go); the metadata guard makes it idempotent.
  • …I need WAL/checkpoint behavior?
    openSQLiteDB DSN pragmas + the postCommit checkpoint closure in db.go.
Gotchas
  • The ledger_entries table was dropped in migration 05 — live entry state is no longer stored locally; getLedgerEntries goes to Core.
  • Reads re-decode the raw LCM blob every time; the transactions/events tables are pure indexes.
  • WAL auto-checkpointing is disabled — checkpointing is manual after every write commit.
  • Two distinct migration mechanisms share this package: SQL schema migrations vs ledger-range data migrations. Don't confuse them.
  • Event id ordering correctness depends on the cursor being a fixed-width zero-padded string — change the format and range scans/trims break.
10

Ingestion & backfill

Network → local store Go

A long-lived loop that writes each closed ledger into SQLite (one DB tx per ledger), with an optional one-shot backfill that bulk-loads history from a cloud datastore first.

The live ingestion goroutine pulls each new LedgerCloseMeta from the ledger backend (Captive Core), and for every ledger writes header + transactions + events into SQLite inside a single atomic transaction, and feeds the transactions into the in-memory fee windows. On startup it resolves where to resume (DB latest+1, or the history-archive tip if empty). The optional backfill phase runs once before live ingestion, bulk-loading historical ledgers in ~960-ledger chunks so the daemon can answer historical queries without waiting for slow live catch-up.

Key abstractions

NameWhat it is
Service structThe engine. Start() launches the run loop; Close() cancels and waits.
Service.ingest funcPer-ledger live path: GetLedger → one WriteTx → write ledger+txs+events → FeeWindows.IngestFees → Commit.
Service.ingestRange funcBackfill bulk path: ONE WriteTx for a whole chunk; does not touch fee windows.
BackfillMeta / RunBackfillVerify gapless → set bounds → backfill backward to retention edge → frontfill forward to datastore tip → re-verify.
One DB tx per ledger (live) vs one per chunk (backfill)
Live commits each ledger atomically (readers see whole ledgers) and checkpoints each time. Backfill batches a ~960-ledger chunk into one commit to amortize commit + checkpoint cost — throughput over latency, since nothing queries mid-backfill.
Fresh backend per backward chunk; persistent for forward
The cloud-datastore backend is monotonic (forward-only). Backward filling needs a brand-new backend per older chunk; forward filling streams through a single backend over one prepared range. This is the entire reason the two paths differ.
ChunkSize ≈ 960 ledgers (OneDay/18)
A RAM/throughput trade-off (~2 GB per chunk) bounding how many LCM blobs are held in one transaction during bulk load.

    Service.ingestLedgerCloseMeta (covers both live + backfill paths).
  • …I need to change startup resume/catch-up?
    Service.getNextLedgerSequence / Service.run.
  • …I need to change backfill ranges / chunk size?
    BackfillMeta.setBounds · ChunkSize const · makeBackend buffer sizing.
Gotchas
  • The live path updates fee windows; the backfill path does not — backfill relies on feewindows.Reset() afterward to repopulate from the DB. If you add per-ledger work, decide deliberately which path(s) need it.
  • RunBackfill refuses to run if the local DB has gaps, and errors if the datastore tip is older than the DB minimum.
  • Frontfill runs twice when backfill was skipped, to catch the tip that advanced during the first pass.
  • After backfill, both db.ResetCache() and feewindows.Reset() run so caches re-derive from the freshly backfilled DB.

Fee windows & request limiters

A ring buffer for fees + two limiters per handler Go

A generic circular buffer of recent ledgers powers classic/Soroban fee-stat percentiles; two composable per-handler limiters enforce throttling and observability on every RPC call.

The fee-stats half keeps a sliding window of the most recent N ledgers' fees entirely in memory (no DB query on the hot path) and recomputes a distribution (min/max/mode + percentiles) on every new ledger. It maintains two independent windows — classic per-operation fees and Soroban inclusion fees — because the two classes price completely differently. The middleware half wraps each handler so a slow or flooded method can't exhaust the daemon.

Key abstractions

NameWhat it is
LedgerBucketWindow[T] generic structThe reusable sliding-window primitive: a slice preallocated to retentionWindow + a start head index; Append evicts the oldest when full and asserts contiguity.
FeeWindows / FeeWindowTwo windows over []uint64 fee lists; IngestFees classifies each tx and appends; GetFeeDistribution returns the cached stats under an RLock.
backlogQLimiterAtomic pending-counter concurrency cap; rejects with 503 / jrpc error over the limit.
requestDurationLimiterRuns the handler in a goroutine with warn + limit timers; aborts with 504 / jrpc -32001 on the limit.
Circular buffer for the ledger window
Ledgers arrive strictly in order and the window only cares about the most recent N. A ring buffer gives O(1) append-with-eviction and fixed memory — no reallocation or shifting as the window slides. Append also asserts contiguity since the source is a monotonic ledger stream.
Two separate fee windows (classic vs Soroban)
Classic fee = feeCharged/numOps; Soroban inclusion fee = feeCharged minus the refundable+non-refundable resource fees from the meta. Separating them lets getFeeStats report independent distributions clients can act on.
Buffered response writer for HTTP timeouts
The downstream handler races a limit timer in a goroutine; its output is buffered so it can be discarded (→ 504) or flushed verbatim — you can't un-write bytes already sent.
Where do I look when…
  • …I need to change fee-stat computation or add a percentile?
    computeFeeDistribution + FeeDistribution in feewindow.go.
  • …I need to change classic-vs-Soroban classification?
    FeeWindows.IngestFees (the op-type switch + meta handling).
  • …I need to adjust per-method or global limits?
    Handler table in jsonrpc.go; thresholds from cfg.RequestBacklog* / cfg.Max*ExecutionDuration.
  • …I want to reuse the sliding window for new data?
    NewLedgerBucketWindow[YourType].
Gotchas
  • The ring buffer requires strictly contiguous ledger sequences — gaps are a hard error, not silently tolerated.
  • computeFeeDistribution recomputes from scratch (flatten + sort) on every ingested ledger — not incremental.
  • Only single-op Soroban transactions count as Soroban inclusion fees; everything else is treated as classic.
  • Limiters are pass-through via sentinels (MaxUint64 / max duration), not “0 means off.”
  • The fee window is populated by being registered as a fake db.Migration — it rides the startup replay, but it is not a schema migration (its Commit is a no-op).

Datastore · xdr2json · util

Cold storage + small shared helpers Go + cgo

Serves historical ledgers from cloud cold storage when local SQLite has aged them out, plus the cgo XDR→JSON converter and the panic-handling goroutine helper.

Three loosely-related support pieces. rpcdatastore lets getLedgers reach past local retention, pulling historical LCM from a remote object store (S3/GCS) via a BufferedStorageBackend. ledgerentries is the live-Core read path for getLedgerEntries/simulateTransaction, surfacing TTL/archival state. xdr2json is the thin Go wrapper over the Rust libxdr2json.a. util.panicgroup decides whether an unhandled panic crashes the process or is just logged.

Key abstractions

NameWhat it is
rpcdatastore.LedgerReaderRead-only access to historical ledgers in object storage. Same name as db.LedgerReader but a distinct type — passed in as DataStoreLedgerReader.

  • GetAvailableLedgerRange is currently hardcoded to FirstLedger: 2 and doesn't query the datastore (a TODO).
  • getLedgers still requires a non-empty local DB even in datastore mode (datastore-only mode is a TODO).
  • xdr2json links the same Rust profile as preflight; a bare go build fails until make build-libs.

Preflight — the Go FFI side

The Go half of the cgo bridge Go ⇄ cgo

Drives Soroban simulation in Rust over cgo, on a bounded worker pool, letting Rust call back into Go to fetch ledger entries on demand instead of copying full ledger state across the boundary.

preflight.go marshals the operation, footprint, source account, and ledger context into C structs, links the platform-specific libpreflight.a via #cgo directives, and invokes the Rust entry points. Critically, rather than serializing the entire ledger state to Rust, it passes a cgo.Handle that Rust uses to call back into Go via the exported SnapshotSourceGet — lazily fetching only the entries a simulation actually touches. pool.go wraps it all in a WorkerPool so CPU-heavy cgo calls run on a fixed set of goroutines behind a bounded queue.

Key abstractions

NameWhat it is
WorkerPool / GetPreflightFixed-size goroutine pool draining a buffered channel; a full queue returns ErrPreflightQueueFull via a non-blocking select.
SnapshotSourceGet //export funcThe reverse callback. Rust calls it during simulation with a LedgerKey; Go recovers the handle, fetches the entry from Core, marshals it back (ttl=-1 ⇒ missing).
snapshotSourceHandleBundles the entry getter + ctx + logger; registered via cgo.NewHandle to get a GC-safe opaque uintptr handed to Rust.
#cgo LDFLAGSPer-OS/arch link lines pointing at target/<triple>/release-with-panic-unwind/libpreflight.a.
cgo.Handle (opaque uintptr)
Go pointers can't be passed into C and held across a call (the GC may move/collect them). cgo.NewHandle registers the handle in a process-global table and yields a stable integer safe to hand to Rust and round-trip back on each callback. This is what makes the Rust-calls-Go data-fetch pattern memory-safe.
On-demand pull vs bulk copy
A full ledger snapshot can be huge, but a simulation reads only a handful of entries. Passing a callback handle lets the Rust host request one LedgerKey at a time and Go fetch just that entry — keeping the cgo boundary cheap.
ttl = -1 sentinel
A missing TTL is encoded as -1 and a zero-value entry signals “not found,” avoiding nullable-struct complications over the C ABI.
Where do I look when…
  • …I need to change what data passes into/out of simulation?
    Parameters/Preflight structs in preflight.go + the C structs in lib/preflight.h + Rust shared.rs.
  • …I need to change how entries are fetched during simulation?
    Swap the LedgerEntryGetter impl; SnapshotSourceGet only depends on that interface.
  • …I hit a cgo link error?
    The #cgo LDFLAGS block at the top of preflight.go — run make build-libs first.
Gotchas
  • SnapshotSourceGet is called by Rust during the cgo call — the inversion of the usual Go→C direction, and the core trick of the subsystem.
  • Never pass a raw Go pointer to Rust; handle.Delete() is deferred so the handle lives exactly one simulation.
  • C memory must be freed explicitly in both directions — forgetting leaks across the boundary.
  • The release-with-panic-unwind profile in the link path is deliberate: it lets Rust catch panics and surface them as errors instead of aborting the daemon.

Rust libraries & the prev/curr dual-protocol pattern

★ The most distinctive design in the repo Rust

Three Rust staticlibs run Soroban simulation and XDR→JSON, with a parametric-module hack that mounts the same simulation code against two soroban versions so one binary serves both the current and next protocol.

The network upgrades protocols on a flag-day, so the RPC must simulate against both the outgoing and incoming protocol from a single binary. Rust lacks parametric modules — so the code mounts the same shared.rs at two module paths and binds each to a differently-versioned copy of soroban.

How the trick works

1 · Dependencies, twice

The workspace Cargo.toml uses Cargo's package = rename to pull soroban-env-host in twice: as soroban-env-host-prev and -curr (likewise soroban-simulation). Two protocol versions, co-resident in one static lib.

2 · Same code, two mounts

lib.rs declares #[path = "."] mod curr and mod prev, each re-exporting its own soroban version under neutral names and mounting shared.rs. shared.rs imports soroban via super::, so each copy compiles against a different version.

3 · Dispatch at runtime

The extern "C" entry points read ledger_info.protocol_version: ≤ prev::PROTOCOL → prev path (also handles older), == curr::PROTOCOL → curr, else error. This is how the binary keeps serving through an upgrade.

4 · Bump = move curr→prev

A protocol cutover moves the curr versions into the prev entries and sets a new curr. scripts/update-soroban-env.sh automates the curr bump on the protocol-next branch.

Key abstractions

NameWhat it is
mod curr / mod prevThe twice-mounted modules; each exposes a PROTOCOL const from its soroban INTERFACE_VERSION.
preflight_invoke_hf_op / preflight_footprint_ttl_opThe two #[no_mangle] extern "C" entry points Go calls; both run inside catch_preflight_panic.
catch_preflight_panicRuns each op on a scoped worker thread with a 100 MiB stack and catches panics, converting them to a result error rather than crossing the C boundary.
GoLedgerStorageImplements soroban's SnapshotSource trait so simulation fetches entries on demand by calling back into Go.
CXDR + ffi helpersThe shared lib/ffi crate owns the {*mut u8, len} blob type and the C-string/XDR alloc-free helpers used by both Rust libs.
Twice-mounted shared.rs (parametric-module emulation)
Rust has no parametric modules, but the RPC must link two soroban releases at once. Including the same file at two module paths — each given a different soroban binding — is the workaround. Any function mentioning a soroban type must live in shared.rs as a free function (it's compiled twice); version-agnostic code stays in lib.rs.
CXDR blob currency
Rather than per-type FFI structs, the boundary passes opaque length-prefixed byte buffers in both directions; each side serializes/deserializes XDR with its own soroban version.
100 MiB dedicated thread + catch_unwind
Soroban invocations can recurse deeply (overflowing the ~8 MiB default stack), and an unwinding panic crossing the C boundary is undefined behavior. Each op runs on a scoped 100 MiB-stack thread wrapped in catch_unwind — requiring the custom release-with-panic-unwind profile.
Where do I look when…
  • …I need to bump the current protocol's soroban version?
    Run scripts/update-soroban-env.sh (also wired into update-soroban-env.yml).
  • …I need a full protocol cutover (curr→prev)?
    Workspace Cargo.toml: move curr versions into the prev entries, set new curr. Usually no Rust code change — only adjust per-lane load_network_config if the new curr's signature differs.
  • …I need to add data returned from simulation?
    Add to CPreflightResult (lib.rs) + populate in shared.rs + free in free_preflight_result + read on the Go side.
  • …I need to add an XDR type to JSON conversion?
    lib/xdr2json/src/lib.rs (xdr_to_json); Go caller in xdr2json/conversion.go.
Gotchas
  • The dispatch is asymmetric: prev uses ≤ (handles all older protocols), curr uses ==. Newer-than-curr errors.
  • shared.rs must import soroban types from super::, NOT crate:: — that indirection is the entire mechanism binding each copy to a different version.
  • Any function whose signature mentions a soroban type must be a free function in shared.rs (it's compiled twice; you can't define the same method twice).
  • The build is hard-gated on the release-with-panic-unwind profile (panic='unwind') — a normal release build would abort on panic and risk UB across the boundary.
  • Memory ownership is split: Rust-allocated results are freed by Rust; Go-allocated entries are freed by Go. Freeing across the boundary is undefined.

Build, test & CI

The hybrid build reality Make · Go · Rust

A Makefile-driven Go+Rust build where the Rust static libs MUST be built before any cgo-linked Go build/test; golangci-lint + clippy gates; a Docker-Compose harness that boots real Stellar Core.

The daemon links two Rust static libs (libpreflight.a, libxdr2json.a) over cgo, so a plain go build/go test fails until those libs exist. The Makefile builds the Rust libs first and injects version metadata via ldflags. Because every CI workflow calls the same Makefile targets, understanding the Makefile is understanding CI.

The targets you'll actually use

# build the Rust libs — REQUIRED before any go build/test make build-libs # build the daemon (build-libs + go build with version ldflags) make build # run a single Go test (after build-libs once) go test -run TestName ./cmd/stellar-rpc/internal/methods/... # full test + lint make test # go-test + rust-test make check # rust-check (fmt+clippy) + go-check (golangci-lint)
build-libs before everything
cgo links the archives at target/<arch>/release-with-panic-unwind/; they don't exist until cargo builds them. This is the single most common stumble.
Version via cargo metadata + ldflags
The soroban-env prev/curr version strings are Rust deps Go can't see, so the Makefile parses cargo metadata with jq and injects them via -ldflags -X — surfaced by stellar-rpc version.
Embedded XDR upgrade fixtures
Integration tests apply Soroban network limits from pre-encoded ConfigUpgradeSet XDR blobs (//go:embed docker/upgrades/*.xdr), versioned per protocol (p25/p26) because limit schemas change across protocols.
Two golangci-lint rules that bite: (1) do not import go-stellar-sdk/support/errors — use stdlib errors + fmt.Errorf; (2) no debug prints (fmt.Print*, print, println) — in tests use t.Log().
Integration tests (gated, real Core)

They spin up real Stellar Core + history archive + friendbot via Docker Compose and run a full daemon, gated behind env vars so they never run during a normal go test ./.... The harness (integrationtest/infrastructure/) drives protocol upgrades and Soroban-limit upgrades from embedded XDR fixtures.

STELLAR_RPC_INTEGRATION_TESTS_ENABLED=true \ STELLAR_RPC_INTEGRATION_TESTS_CORE_MAX_SUPPORTED_PROTOCOL=23 \ STELLAR_RPC_INTEGRATION_TESTS_CAPTIVE_CORE_BIN=$(which stellar-core) \ go test -v -failfast ./cmd/stellar-rpc/internal/integrationtest/...
Gotchas
  • A bare go build reports version dev (no ldflags). Fine for local iteration, not releases.
  • The Core docker image is pinned to an immutable tag — captive-core must stay compatible with the containerized Core.
  • On macOS the Makefile passes -mmacosx-version-min=13.0 to align the Go and Rust linkers.
  • docs/ is auto-generated — don't hand-edit it.
  • User-facing changes go in CHANGELOG.md (there's a /changelog skill).

★ Special data structures, collected

These are the design choices driven by the nature of the data — the ones worth understanding before you touch the hot paths. They recur across the deep dives; here they are in one place.

Raw LCM blob = source of truth

A ledger's full content is large XDR, stored once as a BLOB keyed by sequence. Transaction/event tables hold only pointers; full objects are re-derived from the blob at read time. → §9

Event cursor AS the primary key

A zero-padded <TOID>-<event> string so lexicographic order = chronological order. Range scans, ORDER BY, and retention DELETEs are all plain string comparisons. → §7, §9

TOID sentinel event slots

Pre-tx / per-op / post-tx / post-all-txs stages share one cursor scheme via reserved min/max-mask positions — keeping them sortable and trimmable within a single ledger. → §9

Four topic columns + composite indexes

Up to 4 Soroban event topics, each its own indexed column; (topic1, id) / (contract_id, id) composites keep filtered scans in cursor order. → §7, §9

Twice-mounted shared.rs

Emulates parametric modules so one binary links two soroban versions and serves both protocols across a flag-day upgrade. → §14

On-demand snapshot via cgo.Handle

Rust calls back into Go for one LedgerKey at a time, so full ledger state never crosses the FFI boundary. A GC-safe opaque handle makes it sound. → §8, §13

CXDR blob currency

One opaque {ptr, len} byte-buffer type carries every Stellar value across cgo, in both directions — no per-type FFI structs. → §13, §14

Ring buffer fee window

Generic LedgerBucketWindow[T] — O(1) append-with-eviction, fixed memory, asserts contiguity. Two windows (classic vs Soroban) feed cached percentiles. → §11

Two Core clients

Normal (submit/info) vs a fast client on Core's HTTP query port for latency-sensitive entry reads & simulation. → §4

Batched INSERTs

Tuned to SQLite's ~32.7k bind-variable cap: 3,000 tx rows (3 vars), 1,000 event rows (10 vars). → §9

Monotonic backfill backends

The cloud-datastore backend is forward-only, so backward fill creates a fresh backend per chunk while forward fill reuses one. → §10

Restore preamble (dual tx data)

Simulation returns a separate SorobanTransactionData for a prerequisite RestoreFootprint when an entry is archived. → §8

Getting started

A first-day checklist that respects the hybrid build.

  1. Build the Rust libs once: make build-libs. Without this, every go build/go test fails on missing libpreflight.a.
  2. Build the daemon: make build (gets you real version metadata via ldflags).
  3. Read daemon.go's MustNew top-to-bottom — it's the table of contents for the whole system.
  4. Trace one read end-to-end — e.g. getEvents: jsonrpc.go registration → methods/get_events.go → db/event.go.
  5. Trace one simulation — methods/simulate_transaction.go → preflight/pool.go → preflight/preflight.go (cgo) → lib/preflight/shared.rs, watching SnapshotSourceGet call back into Go.
  6. Run unit tests: make go-test. For integration tests, set the gating env vars (§15) and have stellar-core + Docker available.
  7. Before a PR: make check (clippy + golangci-lint). Remember the two lint rules: no support/errors, no debug prints.
One-line mental model. Captive Core feeds an ingestion loop that stores raw ledger blobs + thin indexes in retention-trimmed SQLite; a JSON-RPC server answers reads from that store (and live Core), while a Rust sandbox over cgo simulates Soroban calls by pulling ledger entries back through Go on demand — all from one binary that links two protocol versions of the soroban host at once.
