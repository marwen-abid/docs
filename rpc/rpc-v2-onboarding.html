Stellar RPC V2,
from the inside.

The same JSON-RPC daemon you know from stellar-rpc — submit transactions, query their status, read ledger entries, simulate Soroban calls — plus the reason this repo exists: a new full-history storage engine built to serve the entire chain (60M+ ledgers, 20B+ events) instead of a 7-day retention window.

Two things to internalize first: (1) this is still a Go daemon linking two Rust static libraries over cgo — that hybrid shapes the build; and (2) the full-history engine is a PoC under construction — its packages are real, tested code, but they are not yet wired into the daemon or any RPC method. Knowing what is and isn’t connected is half the onboarding.

01

The 10,000-ft view

The daemon does three jobs today, and this branch is building a fourth:

① Ingest & store (retention window)

A background service pulls each newly-closed ledger from Captive Core and writes it — ledger blob, transaction index, events — into local SQLite, one DB transaction per ledger, trimmed to a retention window on every commit. This is the daemon’s working memory.

② Serve queries

A JSON-RPC server answers reads (getEvents, getLedgers, getTransaction(s), getFeeStats…) from that SQLite store, plus live state lookups (getLedgerEntries) straight from Core, with an optional cloud-datastore fallback for old ledgers.

③ Simulate (the sandbox)

simulateTransaction runs a Soroban host function in a Rust sandbox over cgo, fetching only the ledger entries it touches via a Go callback. Returns the resource footprint, fees, auth, and predicted state changes.

④ ★ Full-history storage engine NEW · not wired yet

A hot/cold storage stack under internal/fullhistory/ + internal/packfile/ + internal/events/ designed to hold and query the complete chain: RocksDB for the live “hot” chunk, immutable packfiles + bitmap/MPHF indexes for everything older. Built bottom-up as libraries; the daemon doesn’t call any of it yet.

The cast of characters

  • Captive Core — a stellar-core subprocess the daemon launches; source of truth for ledgers and live state. Two HTTP clients: normal (submit/info) and “fast” on Core’s query port (ledger-entry reads & simulation).
  • Ingestion service — the loop turning closed ledgers into SQLite rows (and, in the target design, into full-history stores too).
  • SQLite store — WAL mode, manual checkpointing, retention-trimmed. The classic data plane.
  • JSON-RPC handlers — one per method, wrapped in throttling + metrics middleware.
  • Preflight worker pool + Rust libs — the simulation engine behind the cgo boundary.
  • ★ The Chunk — the full-history engine’s organizing unit: 10,000 consecutive ledgers (chunk_id = (ledger_seq − 2) / 10_000), each producing immutable cold artifacts for ledgers, transaction hashes, and events.
  • ★ packfile — the immutable-file primitive everything cold is built on: one I/O to open, one I/O per item lookup, fully concurrent reads.
  • ★ RocksDB stores — hot-tier stores (events, ledgers, txhashes) plus a tiny metastore that coordinates backfill and streaming via durable flags.

04

Daemon lifecycle & configuration

The composition root + one declaration per setting Go

A Cobra CLI (run · version · gen-config-file) whose daemon.MustNew() constructs every subsystem in a fixed order, configured by an Option pattern where one struct literal defines a setting’s flag, env var, TOML key, default, and validation at once.

main.go is a thin CLI shell; the real work is MustNew (build everything, fail-fast via logger.Fatal) and Run (serve until SIGINT/SIGTERM, then a sync.Once-guarded graceful Close). The Daemon struct holds every subsystem and is passed into them as the narrow interfaces.Daemon abstraction (metrics + the two Core clients + core handle) so they stay testable. Configuration loads in strict precedence — defaults → TOML file → env vars → CLI flags — driven by ~60 Option literals in config/options.go; the env var and TOML key auto-derive from the flag name so the three namespaces can never silently disagree.

Key files

  • main.goCobra root: run (default), version, gen-config-file. No full-history command yet.
  • daemon/daemon.goThe Daemon struct, MustNew startup order, HTTP+admin servers, graceful Close(), storage warm-up + migrations.
  • daemon/interfaces/interfaces.goThe decoupling seam: Daemon, Core-client interfaces, NoOpDaemon for tests.
  • config/options.goThe authoritative list of every setting — add/change config here.
  • config/option.go · config/main.goThe Option struct · the Config struct + SetValues/Validate.
  • config/version.goBuild metadata vars overwritten via -ldflags (commit, branch, soroban-env prev/curr versions).

Where do I look when…
  • …I need to add a subsystem or change start order?
    MustNew + a field on the Daemon struct (daemon/daemon.go). This is also where full-history stores would get wired.
  • …I need to add a config setting?
    Field on Config (config/main.go) + an Option literal in cfg.options() (config/options.go). Guard-rail tests fail until wired correctly.
  • …I need a new CLI command (e.g. a backfill runner)?
    Add a cobra.Command in main.go — the backfill design doc specifies stellar-rpc full-history-backfill --config ….
  • …the version string is wrong?
    config/version.go vars + the GOLDFLAGS/jq block at the top of the Makefile.
Gotchas
  • MustNew never returns an error — every failure is fatal (os.Exit). No partial-startup recovery.
  • Backfill mode (cfg.Backfill) requires ServeLedgersFromDatastore=true; validation fails startup otherwise.
  • The fee-window retention must not exceed HistoryRetentionWindow — startup fatally fails if violated.
  • Startup order is load-bearing: backfill runs before live ingestion, then DB cache + fee windows are reset to repopulate from the backfilled DB.
  • Version/commit/soroban-env strings come from -ldflags; a bare go build reports dev.
05

JSON-RPC transport & limiters

The front door Go

An HTTP POST runs through CORS + a 512 KB size cap + global limiters, bridges onto a creachadair/jrpc2 dispatcher, and each method handler is individually wrapped in a backlog-queue limiter, a duration limiter, and Prometheus metrics.

NewJSONRPCHandler in internal/jsonrpc.go is the single assembly point: it declares every supported method, builds each handler with its dependencies, wraps it in the middleware chain, and mounts the result on a jhttp.Bridge. Cross-cutting concerns (back-pressure, timeouts, metrics, CORS, size cap) live here so handlers stay business-logic-only. The enforcement machinery is in internal/network/: an atomic-counter backlogQLimiter (HTTP 503 / jrpc error when over the cap) and a requestDurationLimiter that warns at a threshold and aborts at the limit (504 / jrpc -32001) using a buffered response writer so a slow handler’s half-written body can be discarded.

The 12 registered methods

getHealth · getEvents · getNetwork · getVersionInfo · getLatestLedger · getLedgers · getLedgerEntries · getTransaction · getTransactions · sendTransaction · simulateTransaction · getFeeStats

AllowArray(false) — named params only
Every handler is built via methods.NewHandler, which forces object params. New request fields can then be added without breaking existing clients. This is a hard project rule: new RPC parameters must be named object fields, never positional.
Two limiter layers
Per-method (inside the bridge) and global (wrapping the whole bridge). When debugging throttling, check both: RequestBacklog*QueueLimit / Max*ExecutionDuration per method, plus the global equivalents.
Request flow. POST → CORS → 512 KB cap → global backlog + duration limiters → jhttp.Bridge (method lookup) → metrics decorator → per-method duration limiter → per-method backlog limiter → the handler → response as base64 XDR or (if format:"json") JSON via the xdr2json Rust lib.
Where do I look when…
  • …I need to add a new RPC method?
    Handlers list in NewJSONRPCHandler (jsonrpc.go) + methods/<name>.go + request/response types in go-stellar-sdk protocols/rpc + config limits in options.go.
  • …I need to change concurrency/timeout limits?
    The per-method config values referenced in jsonrpc.go; enforcement in network/backlogQ.go + network/requestdurationlimiter.go.
  • …I need to find a metric?
    The per-method gauges/counters are created in the jsonrpc.go registration loop; admin port serves /metrics. See docs/MONITORING.md for the operator-facing metric names (global_inflight_requests, <method>_inflight_requests, <method>_execution_threshold_warning/limit).
Gotchas
  • Per-method queue limits are per-instance — behind a load balancer each node enforces its own.
  • The 512 KB request cap is separate from XDR decode memory limits inside handlers.
  • The preflight worker-pool queue is a third, independent queue from the simulateTransaction backlog limiter.

07

Simulate & submit

The only write/execute endpoints Go ⇄ Rust

simulateTransaction preflights a Soroban op against current ledger state via the Rust worker pool; sendTransaction forwards a signed envelope to Captive Core and maps its status back.

simulateTransaction validates the envelope (exactly one operation, no memo), gathers ledger context (latest seq, bucket-list size, protocol version), and dispatches to the bounded preflight.WorkerPool. The pool fails fast with ErrPreflightQueueFull when saturated rather than queueing unboundedly. The response assembles the resource footprint, min fee, return value, auth, state diffs, and — when an entry the tx touches is archived — a restore preamble: a separate SorobanTransactionData for the prerequisite RestoreFootprint transaction. sendTransaction is much thinner: decode, hash, forward to Core, translate the status (ERROR / PENDING / DUPLICATE / TRY_AGAIN_LATER).

LedgerEntryDiff → state changes via Before/After presence
Each diff carries optional Before and After bytes; the presence pattern (before-only = deleted, after-only = created, both = updated) encodes the change type without an enum crossing the FFI boundary.
Bounded job-queue worker pool, fail-fast
Simulation runs untrusted contract code on a 100 MiB-stack thread; a fixed channel + fixed workers cap concurrency, and a full queue rejects immediately so back-pressure is visible to clients.
Where do I look when…
  • …I need to change what simulate accepts or returns?
    NewSimulateTransactionHandler / formatResponse in simulate_transaction.go; mirror changes on the Rust side (§9).
  • …I need to change submission status mapping?
    The status switch in send_transaction.go.
  • …I need to tune simulation concurrency?
    PreflightWorkerCount / PreflightWorkerQueueSize config + createPreflightWorkerPool in daemon.go.
Gotchas
  • simulateTransaction reports most failures as a 200 response with the Error field set; sendTransaction uses typed jrpc errors.
  • ExtendFootprintTtl/RestoreFootprint require caller-supplied SorobanTransactionData; only InvokeHostFunction lets simulation compute the footprint.
  • Simulation always runs against the latest ledger’s protocol version and state — there is no historical simulation.
08

SQLite store & live ingestion

Store the blob, index the rest Go

A WAL-mode SQLite store persisting raw LedgerCloseMeta blobs plus thin index tables, written one atomic transaction per ledger by the ingest loop and trimmed to a retention window on every commit.

The storage philosophy: store the raw LCM XDR blob once, index everything else. The transactions table holds only (hash → ledger, application_order) — fee-bumps get two rows, outer + inner hash — and full transactions are re-derived by seeking into the blob at read time. The events table holds one row per event with the TOID-style cursor string as its primary key, the contract id, and four topic columns, so filters compile to indexed equality predicates and retention trims are a string-compare DELETE. ingest/service.go drives the loop: poll captive core → one WriteTx → write ledger + txs + events → feed fee windows → commit (which trims and manually WAL-checkpoints). An optional datastore backfill (ingest/backfill.go) bulk-loads history in ~960-ledger chunks before live ingestion starts. Fee stats live only in memory: two FeeWindow ring buffers (classic vs Soroban inclusion fees) built on the generic LedgerBucketWindow[T] circular buffer.

Special data structures

Raw LCM blob as the single source of truth
Stored once per ledger; transactions/events tables are pure indexes and full objects are re-derived at read time. Minimal storage, re-parse cost on reads.
Event cursor stored AS the primary key
A fixed-width TOID-style string (ledger, tx, op, event) whose lexicographic order equals chronological order — range scans, ORDER BY, and retention DELETEs are plain string comparisons. Sentinel slots (tx=0, tx=max, op=max) give pre-tx / post-tx / post-all events stable positions.
Batched INSERTs tuned to SQLite’s bind-var cap
~32,766 bind variables max: transactions batch at 3,000 rows, events at 1,000.
Write (per ledger). NewTx → InsertLedger (blob) + InsertTransactions (2 rows for fee-bumps) + InsertEvents (cursor ids + topics) → FeeWindows.IngestFees → Commit: trim ledgers/txs/events older than retention → COMMIT → manual WAL checkpoint.
Where do I look when…
  • …I need a schema change?
    New numbered file in db/sqlmigrations/ (auto-embedded, applied at startup).
  • …I need to change retention/trimming?
    The trim functions called from writeTx.Commit (db/db.go + per-table files).
  • …I need to feed a new store per ledger (e.g. full-history)?
    ingestLedgerCloseMeta in ingest/service.go — the single choke point both live and backfill paths go through.
  • …I need to change fee-stat computation?
    computeFeeDistribution + IngestFees in feewindow/feewindow.go.
Gotchas
  • WAL auto-checkpointing is disabled — the checkpoint is manual after every commit. Skip it and the WAL grows unbounded.
  • Fee-bump transactions are indexed under BOTH outer and inner hash — queries and deletes must expect duplicates.
  • The live path updates fee windows; the bulk backfill path does not (windows are reset + rebuilt afterward).
  • Two migration mechanisms share db/: SQL schema migrations vs ledger-range data migrations. Don’t confuse them.
  • The ingest service does not feed the full-history stores yet — no fullhistory imports anywhere in ingest/ or db/.
09

Rust core & the prev/curr dual-protocol pattern

The cgo boundary Rust ⇄ Go

Three Rust crates (preflight, ffi, xdr2json) statically linked into the Go binary; the same simulation code is compiled twice against two soroban-env-host versions and dispatched on protocol version at runtime — unchanged from V1.

The network upgrades protocols on a flag day, so one binary must simulate against both the outgoing and incoming protocol. Rust lacks parametric modules, so lib/preflight/src/lib.rs mounts shared.rs twice — mod prev and mod curr — each bound to a differently-versioned soroban dependency (Cargo’s package = rename trick in the workspace Cargo.toml). The extern "C" entry points dispatch on protocol_version: ≤ prev::PROTOCOL → prev, == curr::PROTOCOL → curr. On the Go side, internal/preflight/preflight.go passes a cgo.Handle so Rust can call back into Go via the exported SnapshotSourceGet, fetching one ledger entry at a time instead of copying ledger state across the boundary. Panics unwind safely thanks to the custom release-with-panic-unwind cargo profile + catch_unwind on a dedicated 100 MiB-stack thread.

Key abstractions

NameWhat it is

Twice-mounted shared.rs
Any function touching a soroban type must live in shared.rs as a free function — it compiles twice, once per version. Imports must come from super::, not crate::; that indirection is the entire mechanism.
cgo.Handle + on-demand pull
Go pointers can’t be held across C calls; the handle is a GC-safe integer that round-trips through Rust, letting simulation lazily fetch only the entries it touches.

★ The full-history engine: chunks, hot & cold

Why this branch exists NEW Go

A storage and indexing layer designed to serve filtered queries over the complete chain — every ledger, every transaction hash, every event — with p99 ≤ 500 ms, instead of a 7-day retention window.

The Stellar network has emitted 30B+ events over 60M+ ledgers. The classic SQLite plane can’t hold that; this engine can, at roughly ~60 bytes per stored event (~1.3 TB for 22B contract+system events). It is organized around one unit: the Chunk — 10,000 consecutive ledgers, chunk_id = (ledger_seq − 2) / 10_000 (the chain starts at ledger 2), hardcoded in fullhistory/pkg/chunk/chunk.go. Because queries are also capped at 10,000 ledgers, any query touches at most 2 chunks. On disk, chunk files group into buckets of 1,000 (bucket_id = chunk_id / 1000, a pure filesystem concern).

Every data domain has a hot tier (the one mutable chunk receiving live writes, in RocksDB, indexes in memory) and a cold tier (immutable files, indexes on disk). When the hot chunk fills, a freeze turns it into cold artifacts; during bulk backfill, cold chunks are written directly, skipping hot entirely. A RocksDB metastore records which chunk produced which artifact, so backfill and streaming can coordinate without overlap.

★ packfile · intpack · zstd — the immutable storage primitive

One I/O to open, one I/O per lookup NEW Go + cgo

An append-once, sealed-forever file of variable-length byte items, addressable by position, engineered so that opening costs one read and any item lookup costs one more — with unlimited concurrent readers.

Everything cold in the full-history engine sits on this library (internal/packfile/, ~4.7k LOC + 2.5k LOC of tests). The core idea: group items into records (default 128 items each), optionally transform each record through a caller-supplied codec (zstd in practice), and end the file with a compact offset index mapping record → byte position. On open, one speculative 256 KiB read at the tail usually captures trailer + app data + index in a single I/O; the index stays in memory, so every later lookup is an array access + one pread + decode. The design target is storage with constrained I/O budgets (EBS IOPS, S3 range GETs) — minimizing operations matters more than minimizing bytes.

The offset index doesn’t store offsets — it stores record sizes, compressed with Frame-of-Reference (FOR) encoding from internal/intpack/: groups of 128 values, each group bit-packed at the minimum width after subtracting the group minimum (~1–2 bytes per record in practice instead of 8). Multi-item records additionally carry a FOR-encoded item-size index so the reader can slice one item out of a decoded payload. internal/zstd/ binds the system libzstd over cgo (no vendored code) and enforces ≥ 1.5.7 at runtime.

Key files

  • packfile/writer.goWriter: Create → AppendItem → Finish(appData) → Close. Parallel encode/hash pipeline when a codec or content hash is enabled.
  • packfile/reader.goReader: non-blocking Open, ReadItem / ReadItems / ReadRange, Trailer, AppData, Verify. Hot fields laid out cache-consciously.
  • packfile/packfile.goThe 76-byte trailer (magic 0x48434C53, version, format, counts, index size, content hash, CRC32C).
  • packfile/index.go · packfile/record.goOffset-index encode/decode + structural invariant checks · per-record decode workspace (pooled).
  • packfile/codec.go · packfile/contenthash.goRecordEncoder/RecordDecoder interfaces · chunked SHA-256 content hash.
  • intpack/intpack.goEncodeGroup/DecodeGroup: FOR codec with a self-describing 5-byte footer (1B width + 4B min).
  • zstd/zstd.goCompressor (per-worker, NOT concurrent-safe) · Decompressor (process-wide, pools ZSTD_DCtx). Runtime version check.

The API contracts that matter

ContractRule
Borrowed slicesEvery []byte handed to a read callback (or yielded by ReadRange) is valid only inside the callback / loop iteration — the buffer returns to a pool. Copy anything you keep.
Close is mandatoryReader.Close() must always be called (file handle); it is NOT safe concurrently with in-flight reads. Writer.Close() without Finish deletes the incomplete file.
Non-blocking OpenOpen never fails — I/O runs in a background goroutine and errors surface on the first read call.
ReadItems inputPositions must be strictly sorted ascending, no duplicates; with Concurrency > 1 the callback runs from multiple goroutines in arbitrary order.
Codec symmetryEncoder: fresh instance per worker (stateful). Decoder: one shared concurrent-safe instance per process. The trailer’s caller-assigned Format is how readers pick the right decoder.

Special data structures

FOR-compressed offset index of record sizes
Sizes stay small regardless of file size (a 50 GB file of 20 KB records still needs ~15-bit values), so the index for millions of items fits in one I/O and a few hundred KB of memory. Decoded once into a flat int64 offset table.
ItemsPerRecord — the one tuning knob
Bigger records → smaller index + better compression context, but each point read decodes a whole record. ItemsPerRecord=1 inverts the tradeoff. Events use 128; cold ledgers use 1 (each ledger is large and fetched alone).
Layered integrity
CRC32C on the trailer and on the offset index; CRC32C on each item-size index (checked before the expensive decode); zstd frames carry their own xxHash64; optional chunked SHA-256 content hash over the logical item stream for end-to-end Verify.
App-data section
A caller-owned blob between index and trailer — the hook the stores use to embed chunk metadata (ledger offsets for events, first sequence for ledgers) so a chunk file is self-describing. No packfile-level integrity; callers own it.
Point-lookup flow. Open (background: tail read → trailer parse → CRC checks → index decode) → ReadItem(i): record = i / itemsPerRecord → offset table lookup → one pread → strip + CRC-check item-size index → decoder → slice item → callback with borrowed bytes.
Where do I look when…
  • …I need a new record encoding/compression?
    Implement RecordEncoder/RecordDecoder (codec.go), assign a new Format value, plug into Writer/ReaderOptions. zstd/zstd.go is the model.
  • …I need to tune index size vs read granularity?
    WriterOptions.ItemsPerRecord; tradeoff analysis in design-docs/packfile-library.md.
  • …I need faster scattered reads?
    ReaderOptions.Concurrency — fans ReadItems batches out across goroutines, coalescing consecutive records ≤ 1 MiB into single preads.
  • …I’m debugging a corrupt-file error?

Gotchas
  • The #1 bug class: retaining a borrowed slice past the callback. The buffer is pooled — you’ll read someone else’s data later, not crash.
  • Open succeeding tells you nothing — validation errors arrive on the first read.
  • ContentHashExtract must match between writer and reader or Verify fails on an intact file.
  • zstd version is checked at runtime on first use: libzstd < 1.5.7 panics at startup, not compile time. brew install zstd / build from scripts/install-zstd.sh.
  • Reader.Close doesn’t close your decoder — its lifecycle is yours (one process-wide zstd.NewDecompressor() is the intended pattern).

★ Cold events segment — bitmaps, MPHF, post-filter

The most interesting data structures in the repo NEW Go

A frozen chunk’s events become three files — compressed payloads, serialized roaring bitmaps, and a minimal-perfect-hash index — that together answer “which events match contract X and topics Y?” in a handful of I/Os.

The domain types live in internal/events/; the cold writer/reader/index live in fullhistory/pkg/stores/eventstore/. Per chunk:

FileContentsPackfile format
{chunk}-events.packEvent Payloads in order, zstd-compressed, 128/record; ledger-offset array embedded as app data0xFE1E000C
{chunk}-index.packOne serialized roaring bitmap per term, each prefixed with a 4-byte fingerprint, in MPHF slot order, 128/record, uncompressed0xFE1E000B
{chunk}-index.hashMPHF (streamhash) mapping term key → slot in index.pack—

The indexing scheme, in five ideas

① Sequential chunk-local event IDs
Every event in a chunk gets an ID 0…N−1. Dense 32-bit IDs keep roaring bitmaps compact; TOID-style 64-bit IDs would scatter the ID space and bloat every bitmap. The cost: you need a translation from ledger numbers to ID ranges —
② The LedgerOffsets cumulative array
events.LedgerOffsets stores the running event total through each ledger (~80 KB per chunk), giving O(1) ledger → [start,end) event-ID range. It rides inside events.pack as app data, so a cold chunk is self-describing.
③ TermKey: 16-byte hash of (value + field tag)
Five fields are indexed — contractId, topic0–3. Each unique (field, value) pair hashes (xxh3-128, via streamhash’s PreHashInPlace) into a fixed 16-byte key used both as the in-memory map key and, unchanged, as the MPHF input. The field tag in the hash input keeps the same value in different fields distinct.
④ Roaring bitmaps per term
Term cardinality is wildly skewed — most terms match a handful of events, “transfer” matches millions. Roaring picks array containers for sparse terms and compressed bitmap containers for dense ones; query evaluation is bitmap AND/OR. The in-memory memBitmaps goes further: terms with ≤ 64 events use a plain uint32 slice and only promote to a real bitmap beyond that, cutting hot-index memory.
⑤ MPHF + fingerprint + post-filter (the marquee gotcha)
A minimal perfect hash maps each known key to a unique slot — but it maps unknown keys to valid slots too. Defense in depth: each index.pack record is prefixed with the term key’s first 4 bytes (rejects almost all false positives), and because 4 bytes can still collide, query results must always be post-filtered against the actual event bytes. Skipping the post-filter is a correctness bug, not an optimization.

Key files & abstractions

  • events/payload.goPayload — the canonical wire format (61-byte header — version byte 0x01, txHash, ledger/tx/op/event indices, closedAt, XDR length prefix — followed by the ContractEvent XDR). Identical bytes in hot RocksDB and cold packfiles: the freeze pipeline never re-serializes.
  • events/index.goTermKey, Field enum, ComputeTermKey, TermsFor (event → its index terms), BitmapIndex interface.
  • events/membitmaps.gomemBitmaps — concurrent in-memory BitmapIndex with the sparse-list→roaring promotion; Get returns clones, All iterates live pointers under RLock.
  • events/ledgeroffsets.goLedgerOffsets — cumulative counts, single-writer/multi-reader, EventIDs(ledger) range lookup.
  • events/ingest.goLCMToPayloads — walks a LedgerCloseMeta and emits Payloads in canonical order (shared by hot ingest and backfill).
  • stores/eventstore/cold_writer.goColdWriter — streams Payloads into events.pack; Finish(offsets) embeds the ledger-offset array.
  • stores/eventstore/cold_index.goWriteColdIndex — builds the MPHF from a BitmapIndex, writes fingerprint-prefixed bitmaps in slot order, writes index.hash.
  • stores/eventstore/cold_reader.goColdReader — lazy open, background MPHF load, Lookup/LookupKeys/FetchEvents/All.
  • stores/eventstore/cold_format.goFormat IDs, items-per-record constants, codec constructors, ledger-offsets wire codec, MPHF wrapper, per-chunk filename helpers (EventsPackName et al.).
  • stores/eventstore/reader.goThe unified Reader interface — implemented by BOTH the hot store and ColdReader, so the future query coordinator never branches on tier.
Cold query flow. open ColdReader (lazy) → Lookup/LookupKeys: MPHF maps term key → slot → read index.pack record → fingerprint check (mismatch = term absent) → deserialize bitmap → AND/OR bitmaps per the filter → intersect with the ledger range’s event-ID window (via LedgerOffsets) → FetchEvents(sortedIDs): coalesced scattered reads from events.pack → unmarshal Payloads → post-filter: re-verify every term against the real event → results.
Cold write flow (freeze or backfill). for each ledger: LCMToPayloads → ColdWriter.Append (zstd pipeline) + TermsFor → memBitmaps.AddTo + LedgerOffsets.Append → at chunk end: ColdWriter.Finish(offsets) → WriteColdIndex(bitmaps) → fsync all three files → only then flag the chunk complete in the metastore.
Where do I look when…
  • …I need to add an indexed field?
    Field enum + TermsFor in events/index.go.
  • …I need to evolve the event wire format?
    events/payload.go — bump the version byte and keep Unmarshal backward-compatible; frozen chunks live forever.
  • …I need to change a cold file format?
    stores/eventstore/cold_format.go — bump the packfile Format ID / LedgerOffsets version byte; readers reject unknown versions.
  • …I need to swap or tune the MPHF?
    buildMPHF / openMPHF / Lookup in cold_format.go (wraps github.com/tamirms/streamhash) + WriteColdIndex.
  • …I’m implementing the query coordinator?
    Code against eventstore.Reader; post-filtering is the coordinator’s job, not the reader’s.
Gotchas
  • Always post-filter. MPHF + 4-byte fingerprint can pass an event that doesn’t actually match. The reader screens fingerprints; final term verification is the caller’s responsibility.
  • BitmapIndex.All() yields live bitmap pointers valid only inside the loop (read lock held) — marshal inside the loop, never retain.
  • ComputeTermKey must stay bit-identical to streamhash’s internal hashing — any drift silently breaks every existing index.hash.
  • An empty chunk can’t build an MPHF (ErrEmptyBuildSet) — orchestrators must skip WriteColdIndex when no events were written.
  • Mark a chunk complete only after all three files are fsynced — partial artifacts must be discarded and rebuilt on crash.
  • ColdReader.Close is not safe concurrently with in-flight reads (inherits the packfile rule).

★ RocksDB hot stores & the metastore

The mutable chunk NEW Go + grocksdb

A two-layer design: a strict layer-1 RocksDB wrapper that pins durability and lifecycle rules, and layer-2 store facades (events, ledgers, txhashes, meta) that each own their column-family layout and in-memory caches.

Layer 1 — fullhistory/pkg/rocksdb/ wraps grocksdb so no store touches the C API directly. It pins the non-negotiables: WAL always on, every write synced, idempotent CAS-guarded Close that waits for in-flight calls, validated configs, per-CF tuning hooks. Keys that encode integers use big-endian (encoding.go) so byte-lexicographic order equals numeric order and range scans just work. Atomic multi-key commits go through Batch(fn), whose BatchWriter is deliberately inert after the callback returns — a captured reference silently no-ops instead of corrupting a reused handle.

Layer 2 — fullhistory/pkg/stores/:

The four stores

StoreLayoutThe point
eventstore.HotStore3 CFs per chunk: data (eventID → Payload), index ((term, eventID) entries), offsets (ledger → count)Ingests a ledger’s events atomically, then updates the in-memory roaring-bitmap mirror + LedgerOffsets. On restart, warmup rebuilds both mirrors by scanning the index + offsets CFs — RocksDB is the durable truth, memory is a cache.
ledger.HotStoreseq → zstd-compressed LCM bytesSequential workload, no index needed. FirstSeq/LastSeq expose the held range; IterateLedgers streams decompressed bytes (borrowed per iteration). Cold counterpart: ledger.ColdWriter/ColdReader — one packfile per chunk, one ledger per record, firstSeq in app data, so lookup is pure arithmetic with no index at all.
txhash.HotStore16 CFs (cf-0…cf-f) routed by txhash[0] >> 4Pure point-lookup workload (hash → ledger seq). Sharding by high nibble keeps each CF’s bloom filter small and mirrors the planned cold RecSplit index layout exactly, so hot and cold route identically.
metastore.Storeplain string KV in the default CFThe coordination point. Backfill and streaming both consult flags like chunk:00000042:lfs / :txhash / :events and index:00000000:txhash; a flag is written only after the artifact it describes is fsynced — key present ⇒ file durable.

Special data structures

Big-endian integer keys
RocksDB iterates byte-lexicographically; big-endian uint32/64 makes that numeric order for free — no custom comparators, and FirstKey/LastKey give the live range.
Per-CF tuning per access pattern
Event data (sequential write, scattered batch read), index entries (write-heavy, scanned once at freeze), txhash (point lookups, bloom-filter-heavy, auto-compaction off): one DB, different physics per CF.
Durable index entries + in-memory mirror
Bitmaps mutated in place per ledger (~4k adds) would be far too slow if (de)serialized through the DB each time. So: bitmaps live in memory, the (term, eventID) entries are what RocksDB persists, and a restart replays them. Crash recovery and fast ingest from one design.
Hot ingest flow (events). IngestLedgerEvents: validate seq is exactly next-expected → marshal Payloads + derive term keys before any write → one Batch {data + index-entry + offset puts} → synced commit → only then mutate the in-memory mirror + offsets. Rejections leave zero state behind; re-applying a ledger is idempotent.

  • …I need to tune RocksDB (cache, memtables, compaction)?
    Layer-1 rocksdb.Tuning/Config.PerCFOptions + each store’s CF-options function (e.g. hot_store.go in eventstore/txhash).
  • …I need another hot store?
    Copy the facade pattern: own CF layout + encoding in a new stores/<name>/, build on rocksdb.Store, reuse stores/errors.go sentinels.
  • …I need to understand crash recovery?
    warmup* in eventstore/hot_store.go (mirror rebuild) + the metastore key lifecycle in full-history/design-docs/03-backfill-workflow.md.
  • …I’m wiring streaming ingestion?
    The seam is ingestLedgerCloseMeta (ingest/service.go) on one side and HotStore.IngestLedgerEvents / ledger.HotStore.AddLedgers / txhash.HotStore.AddEntries on the other — not yet connected.
Gotchas
  • BatchWriter is dead after the callback — calls on a captured reference are silently dropped, by design.
  • Iterators yield borrowed key/value slices valid only until the next step — same rule as packfile, same bug class.
  • WAL + synced writes are pinned in layer 1 — don’t “optimize” them off; every recovery property depends on them.
  • Ledger ingest is strictly next-expected-sequence; out-of-order writes are rejected before any I/O.
  • Changing the txhash nibble routing after data exists silently misroutes every existing entry.
  • Metastore values are "1" for readability; key presence is the actual signal.

★ Backfill & chunk lifecycle design stage

How history gets in, and how hot becomes cold DESIGN + PARTIAL

An offline full-history-backfill command (designed, not yet implemented) that writes cold chunks directly from archived ledger data as a DAG of idempotent tasks, and a freeze process that turns the live hot chunk into cold artifacts.

The authoritative spec is full-history/design-docs/03-backfill-workflow.md — read it top to bottom before touching this area. The essentials a new developer should hold:

Backfill (design)

  • Offline + idempotent. Backfill ingests a ledger range with no live queries, writing immutable files directly (no hot stores). On failure you simply re-run the command — completed work is never repeated.
  • A DAG of three task types. process_chunk(chunk_id) (no deps; produces ledger pack + raw txhash .bin + events cold segment, checking each output’s metastore flag independently) → build_txhash_index(index_id) (after all 1,000 chunks of a txhash index; builds 16 RecSplit .idx files) → cleanup_txhash(index_id) (deletes the transient .bin files + their flags). A flat worker pool (default GOMAXPROCS) executes the DAG; each task retries up to --max-retries.
  • Crash recovery is organic. Three invariants do all the work: a metastore key is set only after fsync (key ⇒ durable file); every task checks its own outputs and skips/overwrites; the DAG re-registers all tasks on every startup. No reconciliation phase exists or is needed.
  • Ranges expand outward to chunk boundaries (never inward), and the requested range is validated against what the remote ledger archive (BSB / BufferedStorageBackend) actually has — no silent truncation.
  • Streaming finishes what backfill starts. A partially-filled txhash index is completed later by streaming mode, coordinating through the same metastore flags.

Freeze (design; cold writers exist)

Freeze flow (target). hot chunk fills → start the next hot chunk immediately → background: stream All() from the hot store → ColdWriter (events.pack) + WriteColdIndex (index.pack + index.hash) + cold ledger pack → fsync → mark frozen in the metastore in one commit → queries flip to the ColdReader → drop the old hot chunk’s RocksDB + in-memory index. The old hot chunk keeps serving reads throughout; a crash before the final commit restarts the freeze from scratch.

Directory layout (shared by backfill and freeze)

{DATA_DIR}/
├── meta/rocksdb/ ← metastore (WAL always on)
├── ledgers/00000/00000042.pack ← bucket = chunk_id/1000 (%05d), chunk %08d
├── events/00000/00000042-events.pack · -index.pack · -index.hash
└── txhash/raw/00000/00000042.bin ← transient; deleted after RecSplit build
    txhash/index/00000000/cf-{0..f}.idx ← permanent, one dir per 10M-ledger index
Where do I look when…
  • …I’m implementing the backfill command?
    Spec: full-history/design-docs/03-backfill-workflow.md (config schema, DAG pseudocode, getStatus response). Building blocks all exist: chunk math, metastore, ColdWriter/WriteColdIndex, ledger ColdWriter, packfile.
  • …I need the metastore key schema?
    §“Meta Store Keys” of the backfill doc: chunk:{C:08d}:lfs|txhash|events, index:{N:08d}:txhash; chunk:*:txhash keys are deleted after the index is built.
  • …I need the freeze spec?
    §10 + §12 of design-docs/getevents-full-history-design.md (freeze steps, startup procedure, two-hot-segments overlap).
Gotchas
  • CHUNKS_PER_TXHASH_INDEX defines the data layout — once any data exists it must never change between runs.
  • Each chunk output gets its own flag written independently after its own fsync — a crash can legitimately leave a chunk half-flagged; resume logic must check each flag separately.
  • Backfill never prunes: narrowing the range between runs is safe; completed work outside the new range is left alone.
  • A RecSplit verify mismatch is data corruption — abort and investigate, never retry past it.

★ Special data structures, collected

The design choices driven by the nature of the data — append-only ledgers, wildly skewed term cardinality, I/O-budgeted storage. Internalize these before touching the hot paths.

Packfile + FOR offset index

Immutable variable-length items; the end-of-file index stores FOR-compressed record sizes (~1–2 B/record), loaded in one I/O so every lookup is one more I/O. The substrate of all cold storage. → §11

MPHF + fingerprint + post-filter

Minimal perfect hash maps term → slot in O(1) with no stored table — but maps unknown keys somewhere too. 4-byte fingerprints catch most false positives; a mandatory post-filter catches the rest. → §12

Roaring bitmaps, sparse-promoted

Term → event-ID sets where cardinality spans 1 to millions; roaring picks array vs bitmap containers per range, and the in-memory index keeps ≤64-event terms as plain uint32 slices before promoting. → §12

Sequential chunk-local event IDs + LedgerOffsets

Dense 0…N−1 IDs keep bitmaps small; an ~80 KB cumulative-count array gives O(1) ledger → ID-range translation, embedded in the packfile so cold chunks self-describe. → §12

16-way txhash nibble sharding

Point lookups routed by hash[0] >> 4 into 16 column families — smaller bloom filters per CF, and hot routing mirrors the planned cold RecSplit layout exactly. → §13

Durable index entries + in-memory bitmap mirror

RocksDB persists (term, eventID) entries; the queryable bitmaps live only in memory and are rebuilt by replay on restart. Fast ingest and crash recovery from one mechanism. → §13

Big-endian keys + typed chunk.ID

Byte-lex order = numeric order for RocksDB range scans; a distinct ID type makes chunk-vs-ledger mixups a compile error. → §10, §13

Metastore flags: key ⇒ durable file

A flag is written only after its artifact is fsynced, every task checks its own flags, the DAG re-registers everything on startup — crash recovery with no recovery phase. → §13, §14

Raw LCM blob = source of truth

SQLite stores each ledger once as a blob; transactions/events tables are thin indexes and full objects are re-derived at read time. → §8

Event cursor AS the primary key

A fixed-width TOID-style string whose lexicographic order is chronological order — range scans and retention trims are string compares. Sentinel slots cover pre/post-tx events. → §6, §8

Ring-buffer fee windows

Generic LedgerBucketWindow[T]: O(1) append-with-eviction, fixed memory, contiguity-asserted; two windows (classic vs Soroban) feed cached percentiles. → §8

Twice-mounted shared.rs + cgo.Handle callback

One binary, two soroban versions, runtime protocol dispatch; and simulation pulls ledger entries back through Go one key at a time via a GC-safe handle. → §9

The one ownership rule that spans the whole repo: borrowed slices. Packfile read callbacks, ReadRange iterations, RocksDB iterators, and BitmapIndex.All() all hand you memory that is only valid right now — inside the callback, until the next iteration, under the lock. Copy what you keep. Most subtle bugs in this codebase are this rule, violated.

Getting started

A first-week checklist that respects the hybrid build and the PoC nature of the branch.

  1. Native deps once: brew install zstd rocksdb jq (macOS). zstd must be ≥ 1.5.7 — it’s checked at runtime.
  2. Build the Rust libs: make build-libs. Without this every go build/go test fails at link time.
  3. Build and test: make build, then make go-test.
  4. Read the two design docs end to end: design-docs/getevents-full-history-design.md then design-docs/packfile-library.md. They are current and the code matches them closely.
  5. Read daemon.MustNew (daemon/daemon.go) top to bottom — the table of contents for everything that’s wired today.
  6. Trace one classic read — getEvents: jsonrpc.go registration → methods/get_events.go → db/event.go.
  7. Trace one full-history path through its tests — e.g. fullhistory/pkg/stores/eventstore/: hot_store_test (ingest + warmup) then cold_reader/cold_writer tests (freeze-shaped round trip). Tests are the only callers today, so they’re the best executable documentation.
  8. Before a PR: make check; remember the lint rules (no debug prints, no support/errors, 100-line functions) and add user-facing changes to CHANGELOG.md.
