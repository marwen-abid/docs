Benchmarking RPC v2 — what the rest of the world figured out

Field research for the “automated, repeatable benchmarking” problem (roadmap risk #2). We looked at how database vendors, compiler teams, and other chains run perf benchmarks in CI — and what measurement science says about the parts we can't engineer away. Companion to benchmarking-problem.html (the problem framing).

①

Nobody merge-gates on wall-time. Per-PR runs are cheap/deterministic; full wall-time suites run nightly with retrospective stats.

②

Pair, don't pin. ClickHouse cancels machine noise by running baseline + candidate on the same box, interleaved — per-PR, on shared CI.

③

v2's I/O budget is assertable. The packfile contract is “1 I/O to open, 1 per lookup” — countable on any runner, no clock needed.

④

Thresholds don't work. MongoDB saw up to 99% false positives; change-point detection over a nightly series is the industry answer.

⑤

p99 needs open-loop load. Closed-loop drivers under-report tails by up to ~200× (coordinated omission) — exactly at our 500 ms SLA percentile.

01The cadence map: what actually runs where

Across every team we examined, perf testing splits into three layers by cost and determinism — no one runs their heavyweight wall-time suite as a blocking per-PR gate. This is the shape #775 should inherit instead of inventing.

PER-PR / PER-MERGEminutes · blocking or auto-comment
ClickHouse ~3,000 query benches on every PR — viable only because of the paired same-machine design (§02)[1]. rustc auto-runs the full suite after every merge on a dedicated collector + on-demand per-PR via @rust-timer queue[4]. CodSpeed users run benches once under CPU simulation on normal shared runners — deterministic, so a 10% default threshold works[8].
RPC v2 ⇒ per-PR gate = deterministic proxy checks (§03) + paired quick query benches (§02). Small, always-on, trustworthy.
NIGHTLY / PERIODIChours · non-blocking · time-series
MongoDB ~200 end-to-end benchmarks daily on real EC2 clusters; per-patch runs are opt-in, never a merge gate[9]. DataStax nightly runs → Graphite → Hunter change-point detection → Slack[11]. Bitcoin Core continuous IBD + microbench runs on dedicated idle hardware (bitcoinperf → codespeed/Grafana)[6].
RPC v2 ⇒ #775's real home: nightly full ingest+query suite on the pinned box, golden data, results appended to the CSV/GCS series, CPD on top (§04).
RELEASE / AD-HOCdays · manual · exploratory
Polkadot re-derives on-chain weights by benchmarking worst-case extrinsics on standardized hardware (10¹² weight ≡ 1 s on the reference box)[7]. NEAR re-estimates gas params in a container under QEMU instruction counting[5]. Stellar (today) the #750 cross-machine sweep is exactly this layer.
RPC v2 ⇒ SLA validation (#776 §4), hardware-recommendation sweeps, v1-vs-v2 comparisons. Run per release or per design milestone, not per change.

02Trick 1 — paired comparison cancels the machine (ClickHouse)

The strongest single idea we found. ClickHouse used to compare against historical numbers and got burned by environment drift (“a machine with a dying HDD”, a broken atop update). Now they run the reference binary and the PR binary simultaneously on the same machine, alternating queries — systematic errors hit both sides equally and cancel[1]. That's what makes per-PR perf testing possible without quiet dedicated hardware.

Disk spikes, noisy neighbors, thermal drift — all hit both distributions. Compare the two samples, not either sample against history.

Per query: median of N runs each side → difference D; a non-parametric bootstrap builds the “nothing changed” distribution → per-query threshold T[1]:
|D| ≤ Tnot statistically significant → pass
|D| ≤ 5%significant but too small to matter → pass
T ≥ 10%the test is too noisy to measure anything → flag the benchmark, not the code
|D| ≥ T and ≥ 5%real, notable change → report with flamegraph + counter diff

For #771: the query benches (cold-*/hot-*) read immutable chunks — two daemon builds can point at the same golden data concurrently with zero state conflict. v2's architecture is unusually friendly to this trick. ClickHouse's two cultural lessons travel too: unstable benchmarks are a first-class defect (several hundred of their queries turned out to measure nothing), and false positives quietly turn the whole system into “mining the green check”[1].

03Trick 2 — deterministic proxy metrics for the per-PR gate

When teams need per-change signal, they stop measuring time and start counting things that don't vary: instructions, simulated cycles, I/O operations. Wall time then only needs to be measured nightly, where noise can be averaged across a series.

I/O operations & bytes

NEAR io-trace★ RPC v2

NEAR replays recorded I/O traces and counts DB GET/SET requests and bytes to ground gas costs[5]. Fully deterministic.

Instructions executed

rustcNEAR icount

rustc-perf's default comparison metric; significance = IQR-fence vs the test's own history[3]. NEAR prices gas via QEMU icount because wall time is “highly variable between runs and physical machines”[5].

Simulated cycles

CodSpeed

cycles ≈ Ir + L1-miss·cost + LL-miss·cost; benchmark runs once, on any shared CI runner, hardware-agnostic[8].

Wall time

everyone, nightly

The only metric users feel — but it needs pinned+tuned hardware, repetition, and series-level statistics (§04, §05).

The v2-specific gem: the packfile library was designed to an I/O budget — one read to open, one pread per item lookup, coalesced scatter reads (design-docs/packfile-library.md). That contract is countable: a per-PR test can open a golden chunk and assert syscall/pread counts, RocksDB perf-context counters, and Go allocation counts for canonical operations. A regression like “lookups now take 2 I/Os” or an accidental O(n) scan is caught deterministically, on any GitHub runner, with no clock involved — the same move as ClickHouse recording perf_event counters per query to explain time changes[1], but used as the gate itself.

04Trick 3 — regressions are change-points in a series, not threshold breaches

MongoDB ran threshold-based detection (“flag if >X% vs last run / last week / last release”) for years — and measured its false-positive rate at up to 99%. Their replacement, E-Divisive change-point detection over the full nightly time series, is now the published industry baseline (and DataStax's Hunter refined it)[10] [11].

Threshold on single runs

Noise crosses the line; a real-but-gradual shift never does. MongoDB: ~100 notifications per 1 real tracked issue[10].

Change-point detection on the series

Compares the distribution before vs after every candidate point; flags the commit window where the level moved. MongoDB: 9 notifications → 4 real issues[10].

≈30 points

Hunter's default window — a month of nightlies. Two weeks (14) was empirically too few for stable detection[11].

≥5% filter

Change-points below 5% magnitude are dropped as not actionable[11] — the same floor ClickHouse uses per-query[1].

~1–6 days lag

CPD is retrospective — it needs ≥3 points after a change. MongoDB flags regressions days after the commit; attribution, not gating[10] [11].

For #775: the per-iteration CSVs already land in gs://rpc-full-history/benchmarks/ — the missing piece is treating them as a time series. Run nightly on the pinned box, append, and run Hunter (open-source, t-test-based, deterministic[11]) over the series with a Slack sink. Accept the lag: the per-PR layer (§02–03) catches the big/deterministic stuff; CPD catches the slow 7% drift no gate ever will. Don't promise blocking detection of small wall-time regressions — nobody we found has it.

05The noise floor is a property of the setup, not the statistics

Three independent results say the same thing: fix the environment first, then add statistics — never the reverse.

±10% → 0 FP

“Statistics cannot rescue bad hardware.” One DataStax team on untuned datacenter boxes saw ±10% median noise and ~3 false change-points/month; the same benchmarks moved to tuned cloud instances → zero false positives[11].

EC2 ≈ on-prem

MongoDB empirically showed tuned EC2 is as repeatable as their on-prem lab, and baked that tuning into their harness defaults so every run inherits it[9]. Validates the fixed-instance plan from #750 — pin one type and document the OS tuning.

33–300%

Mytkowicz et al.: UNIX env size (stack alignment) swung a benchmark 33% routinely, ~300% worst case; link order alone biased results 2–8%[12]. ClickHouse independently sees true-positive perf changes from executable layout with no relevant code change[1].

Practically: normalize env/ASLR in the harness, expect occasional layout-driven “regressions” that bisect to nothing, and budget repetitions where the variance actually lives — Kalibera & Jones: run a one-time dimensioning experiment (variance across machine-provision vs process-run vs query-iteration), then hardcode repetition counts at the noisy level; report effect sizes with confidence intervals (“+5.5% ± 2.5%”), not p-values; determine warmup once per benchmark and hardcode it — automated warmup heuristics pick wrong in both directions[13]. This is exactly the §7.0 methodology slot in #776.

07Golden state: v2's immutability makes the v1 problem mostly disappear

The v1 CI design fights hard for “identical starting DB per run” (EBS volume clones of a golden SQLite volume) because SQLite is one big mutable file. v2's layout changes the economics — and Bitcoin Core has run the resulting pattern for years: bitcoinperf benchmarks initial-block-download over “an interesting range of the chain” on top of a preexisting synced datadir[6] — structurally identical to #750's “ingest chunk 5999 on top of golden chunks 5859–5998”.

v1 — golden state is hard

  • One mutable SQLite file; every run dirties it (tombstones, WAL growth)
  • ⇒ full volume clone per run (EBS clones, <2 min) or slow S3 restore
  • Representative DB = 350+ GB that must be byte-identical every time

v2 — golden state is mostly free

  • Cold chunks are immutable by design — written once at freeze/backfill, never modified. Reads don't dirty them; they can live on a shared read-only volume/snapshot indefinitely
  • Per-run reset = wipe the hot RocksDB dir + metastore (one small directory), re-mount the cold set
  • Ingest benches write new chunk files — delete them after; the golden set is untouched
  • Synthetic datasets (#762) are reproducible from (apply-load config, seed) — the golden set itself can be regenerated, not just restored (NEAR containerizes its estimator for the same reason[5])

Caveat that keeps EBS clones relevant: filesystem cache state and EBS lazy-loading still matter — §7.0's cold-cache eviction methodology and a warm-the-volume step remain necessary, and the ingestion-CI design (golden DB + disposable clone) is still the right shape for any benchmark that mutates shared state. The v1 doc's analysis transfers; it just applies to a much smaller mutable surface.

08Recommendations, mapped to the open issues

In priority order. Each one is a borrowed, working practice — none is invented here.

1
Split the cadence: deterministic per-PR, wall-time nightly
Shape #775 as two jobs, not one: a per-PR gate built from §02/§03 (paired quick benches + counted invariants, minutes, any runner) and a nightly suite on the pinned instance feeding a time series. Per-PR wall-time gating of the full suite is the one design nobody in the field ships.
#775MongoDB · rustc · DataStax
2
Add paired same-machine comparison to the query benches
Teach bench-fullhistory to run two daemon builds against the same golden chunks, interleaved, and emit (D, T) per benchmark with ClickHouse's 4-case verdict. This is the cheapest path to trustworthy per-PR perf signal — no golden hardware required.
#771ClickHouse
3
Assert the I/O budget as a deterministic per-PR test
The packfile/cold-reader contracts (“1 I/O open, 1 pread/lookup, coalesced scatters”) and RocksDB perf-context counters are exact, countable invariants. Assert counts for canonical ops (open chunk, point lookup, 1k-event scatter, ledger ingest) in ordinary CI. Catches algorithmic regressions with zero noise.
#771 · #776rustc · NEAR · CodSpeed
4
Run change-point detection over the nightly series — start accumulating now
Append nightly results to the existing GCS CSV store and run Hunter (or E-Divisive) with a 5% magnitude filter and Slack output. It needs ~30 points to be trustworthy — every week not collecting is a week of delay on #775's value.
#775MongoDB · Hunter
5
Write the §7.0 methodology to kill known measurement bugs
Open-loop constant-rate load + HdrHistogram for every latency scenario (coordinated omission); effect-size CIs not p-values; one-time dimensioning experiment to place repetitions; fixed, hand-determined warmup counts; env/ASLR normalization.
#776wrk2 · Kalibera&Jones · Mytkowicz
6
Exploit immutability for golden state; keep clones only for the mutable surface
Golden cold chunks: one read-only volume/snapshot, shared across runs. Per-run reset: wipe hot RocksDB + metastore, delete bench-written chunks. Keep the v1 EBS-clone pattern for ingestion benches that mutate shared state, plus an explicit cache-eviction/volume-warm step.
#762 · v1 docbitcoinperf · design docs
7
Pin one tuned instance type; treat benchmark instability as a defect
One instance type from the #750 sweep, with documented OS/EBS tuning (MongoDB-style defaults in the harness). Track the false-positive rate of the benchmark system itself, and auto-flag benchmarks whose own variance exceeds ~10% as broken tests (ClickHouse's “unstable query” class) — trust is the product.
